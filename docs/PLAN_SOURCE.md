# DETAILED IMPLEMENTATION PLAN — CLI/TUI AI-ревьюер git-истории

> Рабочее кодовое имя проекта: **`gitl`** (git-log-lens) — короткое, легко набирается, свободно в `go install`-неймспейсе. В плане далее используется как `toolname`/`gitl` взаимозаменяемо.

---

## 1) Обзор

`gitl` — CLI/CI-инструмент на Go: читает git-историю репозитория (коммиты, диффы, статистику за период или диапазон) и через LLM-API превращает её в структурированный инженерный артефакт — AI-ревью PR с машиночитаемым риск-скорингом для гейтинга в CI, changelog, командный дайджест активности. Спроектирован так, чтобы им можно было реально пользоваться в CI/CD-конвейере команды, а не только локально у автора.

Приоритет обучения Go никуда не делся (пользователь явно просил «если Go — то фокус на написании и обучении»), но ambition поднят: вместо «минимального CLI, полезного лично мне» цель — **инструмент, который не стыдно поставить в CI реального проекта и предложить команде**, с соответствующим набором ожиданий по стабильности вывода, безопасности релизов и governance для командного использования. Это не отменяет обучающую нагрузку на стандартную библиотеку (структура проекта, горутины/каналы, `os/exec`, `net/http`-клиент со стримингом и retry, table-driven-тесты, кросс-компиляция) — она остаётся полностью, просто вокруг неё выстроен более серьёзный продуктовый контур.

Ключевые свойства:
- **Нулевая инфраструктура для автора.** CLI-бинарник + GitHub Action-обёртка — сервер/БД/деплой не нужны; конечный пользователь (человек или CI-раннер) сам держит своё окружение.
- **BYOK + мультипровайдерность как продуктовое требование**, не «приятный бонус» — команды с требованиями к резидентности данных должны иметь путь на Ollama/Azure OpenAI/self-hosted endpoint без доработки напильником.
- **Governance для командного, не только личного использования.** Repo-level shareable-конфиг, cost-guard на LLM-расходы, версионированная JSON-схема вывода для интеграции с другими инструментами.
- **Минимум общения от автора.** Вся эта продуктовость — самообслуживанием (GitHub Marketplace, документация, семантическое версионирование), а не через саппорт-чат.

Почему всё ещё хороший «первый» проект портфеля: полный жизненный цикл серьёзного Go-инструмента (CLI-фреймворк → внешние процессы → HTTP-клиент → CI-интеграция → релизный инжиниринг с подписью артефактов) — просто с более высокой планкой готовности, чем «работает у меня на ноутбуке».

## 2) Вводные и ограничения

**Разработчик:**
- Соло, интроверт, реализация с AI-ассистентом (Cursor/Claude Code).
- Приоритетный язык — Go, режим «написание + обучение» (нагружать язык, а не SDK).
- Бюджет: до $500 разово и ≤ $50–70/мес суммарно по всему портфелю; для *этой* идеи целевые инфраструктурные расходы ≈ **$0/мес**.
- Экономическая обоснованность необязательна; побочный доход — опциональный бонус.

**Технические вводные (уже согласованы):**
- CLI-фреймворк — **cobra** (де-факто стандарт: kubectl, gh, hugo).
- Конфиг — **viper** (родной компаньон cobra: YAML-файл + env + флаги в одной иерархии).
- Опциональный TUI-слой — **bubbletea** (Charm) поверх готовых текстовых артефактов, для интерактивного просмотра `digest`.
- Git — через `os/exec` к системному `git` (обоснование — в §4).
- AI — BYOK, OpenAI-совместимый HTTP API (`/v1/chat/completions`), ключ из `~/.config/gitl/config.yaml` или env.
- Дистрибуция — `go install`, GitHub Releases с кросс-компиляцией (Linux/macOS/Windows), опционально Homebrew tap, заявка в `awesome-go`.

**Ограничения и допущения:**
- Требуется установленный системный `git` в `PATH` (для варианта `os/exec`) — это приемлемо, т.к. целевая аудитория — разработчики.
- LLM обрабатывает **текст диффа**, значит есть жёсткий потолок по токенам → нужна стратегия усечения/чанкинга (см. §6).
- Go **1.22+** (нужны `slices`, `maps`, `log/slog`, современный `errors.Join`).
- Кросс-платформенность: пути и конфиг через `os.UserConfigDir()`, а не хардкод `~/.config`.
- Без телеметрии и «звонков домой» — принципиально (доверие + приватность диффов).

## 3) MVP-скоуп

### 3.0 Validation-first спайк (~первая условная неделя)
Прежде чем строить весь CI/governance-контур — снять технический риск №1 (годится ли вообще AI-ревью диффа для реального использования) на одной захардкоженной команде:

> `gitl review HEAD~5..HEAD` — берёт диапазон, гонит `git log` + `git diff`, шлёт в LLM (или в offline-заглушку), печатает Markdown-ревью в stdout.

Это не защита от выгорания, а стандартная инженерная дисциплина: не строить дорогую CI/governance-обвязку вокруг ядра, качество которого не проверено. Если базовое AI-ревью на реальном диффе бесполезно — дешевле узнать это за 2-3 дня на голом прототипе, чем через много недель на полностью собранном продукте. Критерий: вывод осмыслен на реальном репозитории. Если нет — сначала правим промпт/парсинг, потом наращиваем контур.

### 3.1 Must-have (v1 — продуктовый, CI-ready с первого релиза)

**Ядро (AI-анализ):**
- **`gitl review <range>`** — AI-ревью с **машиночитаемым риск-скорингом** (`low|medium|high`), не только текстовым нарративом — чтобы CI мог гейтить (`--fail-on=high` → ненулевой exit code).
- **`gitl changelog --since=<ref>`** — changelog в стиле [Keep a Changelog], группировка по типам через conventional-commits, если распознаются.
- **`gitl digest --days=N`** — сводка активности; **мульти-репо режим** (`--repos=a,b,c` или список в конфиге) — агрегированный дайджест по нескольким сервисам. Это реальный enterprise-сценарий: тимлид/платформенная команда следит не за одним репозиторием.

**CI-интеграция (в MVP, не отложено на потом):**
- **GitHub Action-обёртка** (`action.yml`, composite или Docker action) — публикуется в Marketplace **вместе с v1**. Постит `review` комментарием к PR, выставляет check status по риск-скорингу.
- **Версионированная JSON-схема вывода** (`--format=json` с полем `schema_version`) — документированный контракт для внешних интеграций, не ломающийся молча между релизами.

**Governance для командного использования:**
- **Repo-level конфиг `.gitl.yaml`** в корне репозитория (приоритет: флаг > env > `.gitl.yaml` в репо > личный `~/.config/gitl/config.yaml`) — команда коммитит одну политику на всех: исключённые пути, обязательные changelog-категории, порог риска для гейтинга.
- **Cost-guard**: `--dry-run` оценивает стоимость вызова по токенам ДО реального запроса; `--max-cost-usd` отказывает выполнять запрос дороже порога. Обязательно для CI-сценария — никто не хочет непредсказуемый LLM-счёт от каждого PR.
- **Мультипровайдерность как контракт, не побочный эффект `base_url`**: явно протестированные референс-конфигурации для OpenAI-совместимого API, **Ollama** (локально/self-hosted) и **Azure OpenAI** (data residency для enterprise) — три сценария в документации и в интеграционных тестах, не «наверное сработает».

**Базовое:**
- Конфиг + env-override, флаги вывода (`--format=md|text|json`, `--output=<file>`), offline-режим (детерминированная эвристика без сети — используется и для dev/CI, и как fallback без ключа), ретраи/таймауты (экспоненциальный backoff + jitter, context-cancel по Ctrl-C).
- **`gitl version`** с build-provenance (commit hash + дата сборки через `-ldflags`), а не просто номер версии — важно для саппорта/дебага у пользователей.

**Релизный инжиниринг (в MVP, не «когда-нибудь»):**
- **Подписанные релизы** (`goreleaser` + `cosign`/Sigstore keyless signing) + `SHA256SUMS`. Supply-chain-доверие — отсутствие подписи часто прямая причина, по которой команды не пускают сторонний CLI в свой CI.

### 3.2 Nice-to-have (после v1)
- Стриминг ответа LLM в терминал (токен-за-токеном).
- Горутинный параллельный сбор `git diff` по нескольким коммитам/файлам (worker-pool — реальная нагрузка на каналы, §4).
- Кастомные шаблоны промптов/вывода (Go `text/template`).
- `gitl digest --tui` на bubbletea — навигация по коммитам/файлам, свёртка секций.
- Кэш ответов LLM по хэшу диапазона (не переплачивать за повтор) — `~/.cache/gitl/`.
- Полный SLSA provenance (не только подпись, но и верифицируемая цепочка сборки) — следующий шаг после базового cosign-подписания.

### 3.3 Excluded — Never (сознательно вне скоупа)
- Свой веб-UI / SaaS / сервер / хостинг ключей — противоречит «$0 инфра автору» и «минимум общения». Продуктовость — в качестве и governance CLI/Action, не в построении облачного сервиса.
- Мультипользовательские аккаунты, биллинг, подписки внутри самого тула (GitHub Marketplace, если понадобится монетизация Action, берёт это на себя).
- Проксирование LLM через свой бэкенд (это чужие токены + расходы + приватность — конфликтует с BYOK).
- Интеграции с трекерами (Jira/Linear) в v1 — расширение скоупа без ясной обучающей ценности для Go-ядра; кандидат на v2, если появится спрос.
- GUI-приложение (это отдельные проекты портфеля — Tauri/IntelliJ).
- Автопостинг сводок куда-либо, кроме PR-комментария через собственный же GitHub Action (Slack/email-интеграции) — оставляем на усмотрение пользователя через pipe/JSON-вывод в его собственную автоматизацию.

## 4) Технический стек и архитектура

### 4.1 Стек
| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | Go 1.22+ | Приоритет обучения; стандартная библиотека закрывает 90% нужд |
| CLI | `spf13/cobra` | Стандарт индустрии, чистое дерево команд/флагов |
| Конфиг | `spf13/viper` | YAML + env + флаги в единой иерархии, интеграция с cobra |
| Git | системный `git` через `os/exec` | см. §4.3 |
| HTTP к LLM | `net/http` (stdlib) | Осознанно **без** SDK — тренировка ручного клиента, стриминга, retry |
| JSON | `encoding/json` (stdlib) | Парсинг ответов LLM и `--format=json` |
| TUI (opt) | `charmbracelet/bubbletea` + `lipgloss` | Только для `digest --tui`, поверх готовых артефактов |
| Логи | `log/slog` (stdlib) | Структурные логи, `--verbose` → debug-уровень |
| Тесты | stdlib `testing` + table-driven | Идиоматика Go; golden-файлы для рендеров |
| Сборка/релиз | `goreleaser` | Кросс-компиляция + GitHub Releases + Homebrew одним конфигом |

Сознательное ограничение зависимостей: git — stdlib `os/exec`, HTTP — stdlib `net/http`. Внешние зависимости только там, где они дают обучающий каркас (cobra/viper/bubbletea) или несоразмерны для ручной реализации (goreleaser).

### 4.3 `os/exec` vs `go-git` — решение и обоснование
**Выбор: системный `git` через `os/exec`.** Причины, релевантные цели «написание + обучение Go»:
1. **Тренирует ядровой Go-навык** — работу с внешними процессами: `exec.CommandContext`, чтение `stdout`/`stderr` через `bytes.Buffer`/`bufio.Scanner`, коды возврата, контекст-отмена. Это переиспользуемый на будущих проектах паттерн.
2. **Заставляет писать парсер** — самая обучающая часть: спроектировать `--pretty=format:` с разделителями, распарсить `--name-status`, покрыть table-driven-тестами на golden-выводе. С `go-git` данные приходят уже структурами — парсинг «исчезает», а с ним и упражнение.
3. **Простая ментальная модель** — поведение 1:1 с тем, что разработчик набирает в терминале; отлаживается тривиально (скопировал команду, запустил вручную).
4. **`go-git` тяжелее и с нюансами** — не 100% паритет с CLI по краевым случаям, больше зависимость, меньше «ванильности».

Компромисс инкапсулируется: весь `os/exec` спрятан за интерфейсом в `internal/gitlog`, поэтому если позже захочется `go-git` — меняется одна реализация, команды не трогаются.

```go
// internal/gitlog/types.go
type Commit struct {
    Hash    string
    Author  string
    Date    time.Time
    Subject string
    Body    string
    Files   []FileChange
}
type FileChange struct {
    Status string // A, M, D, R...
    Path   string
    Old    string // для переименований (R)
}

// internal/gitlog/runner.go — пример реального вызова
// git log --pretty=format:%H%x1f%an%x1f%aI%x1f%s%x1f%b%x1e --name-status <range>
//   %x1f (unit separator) разделяет поля, %x1e (record separator) — коммиты.
func (r *Runner) Log(ctx context.Context, rng Range) ([]Commit, error) {
    args := []string{"log",
        "--pretty=format:%H%x1f%an%x1f%aI%x1f%s%x1f%b%x1e",
        "--name-status", rng.String()}
    out, err := r.run(ctx, args...) // exec.CommandContext("git", args...)
    if err != nil { return nil, err }
    return parseLog(out) // parser.go, покрыт table-driven тестами
}
```
Управляющие символы `%x1f`/`%x1e` вместо пробелов/переводов строк — потому что тело коммита содержит и то, и другое; разделители из ASCII-control-зоны в сообщениях не встречаются. Это важная деталь надёжного парсинга (частая ошибка новичка — split по `\n`).

### 4.4 Схема конфиг-файла (личный + repo-level)

Два уровня, сливаются по приоритету **флаг > env > `.gitl.yaml` (репозиторий) > `~/.config/gitl/config.yaml` (личный)**. Личный файл — путь через `os.UserConfigDir()` (на Windows → `%AppData%\gitl`); repo-level — `.gitl.yaml` в корне репозитория, коммитится в git и является общей политикой команды.

```yaml
# ~/.config/gitl/config.yaml — личные настройки (секреты, провайдер по умолчанию)
llm:
  provider: "openai"                # "openai" | "ollama" | "azure_openai" — явный контракт, не угадывание по base_url
  api_key: ""                       # или env GITL_API_KEY (рекомендовано, особенно в CI)
  base_url: "https://api.openai.com/v1"
  model: "gpt-4o-mini"
  max_tokens: 1500
  temperature: 0.2
  timeout_seconds: 60
  max_retries: 3
  # azure_openai: endpoint, deployment, api_version — отдельный под-блок при provider=azure_openai
  # ollama: host (по умолчанию http://localhost:11434) — без api_key

cost:
  max_cost_usd: 0.50                # cost-guard: отказ выполнять запрос дороже порога
  warn_at_usd: 0.10                 # предупреждение, но не блокировка

output:
  format: "md"                      # md | text | json
  color: true

diff:
  max_diff_bytes: 120000
  exclude_globs:
    - "*.lock"
    - "*.min.js"
    - "vendor/**"
    - "*.svg"
```

```yaml
# .gitl.yaml — repo-level политика (коммитится в репозиторий, общая для команды)
policy:
  fail_on: "high"                   # low | medium | high | never — порог гейтинга CI
  required_changelog_categories: ["Added", "Fixed", "Security"]
  exclude_globs: ["migrations/**"]  # добавляется к личным exclude_globs, не заменяет их

digest:
  repos:                            # мульти-репо digest, если запущен из "родительской" директории/монорепо-конфига
    - path: "../service-a"
    - path: "../service-b"
```

Валидация при старте: если `api_key` пуст/недоступен → offline-режим с предупреждением в stderr (не падаем). Провайдер выбирается явным полем `provider`, а не угадыванием по `base_url` — это и есть контракт мультипровайдерности из §3.1, а не «наверное сработает».

### 4.5 Поток данных (одна команда)
```
cobra cmd → config.Load() → gitlog.Runner.Log/Diff(ctx, range)
   → prompt.Build(commits, diff, template)
   → llm.Client.Complete(ctx, prompt)  [retry/backoff, offline-fallback]
   → render.Render(artifact, format)   → stdout/file/TUI
```
Отмена по Ctrl-C — единый `context.Context` из `signal.NotifyContext`, протянутый через exec и HTTP.

## 5) Roadmap по неделям (≈10–12 условных блоков эффорта)

> «Неделя» — условный блок, не календарная привязка. Порядок важнее сроков. Роадмап длиннее, чем был бы у минимального личного CLI — это прямое следствие поднятой планки (CI-гейтинг, мультипровайдерность, governance, подписанные релизы не бывают бесплатными по времени, только по деньгам).

### Неделя 1 — Валидационный спайк
- **Фокус:** снять риск №1 — годится ли AI-ревью диффа в принципе, прежде чем строить контур вокруг него.
- **Действия:** `go mod init`; минимальный `cmd/gitl/main.go` + команда `review <range>`; `internal/gitlog` с `Log`+`Diff` через `os/exec`; захардкоженный промпт; примитивный `net/http`-вызов (без retry) + offline-заглушка; вывод в stdout.
- **Deliverable:** `gitl review HEAD~5..HEAD` печатает осмысленное Markdown-ревью на реальном репозитории.
- **Критерий успеха:** вывод содержательный; парсер корректно ловит переименования/удаления файлов.

### Неделя 2 — Каркас CLI, конфиг (личный + repo-level), offline-режим
- **Фокус:** превратить прототип в настоящее CLI-приложение с governance-конфигом с первого дня.
- **Действия:** cobra (root + подкоманды) и viper; `internal/config` со struct + валидацией + слиянием `.gitl.yaml` (репо) → `~/.config/gitl/config.yaml` (личный) → env → флаги; глобальные флаги; `log/slog`; детерминированный offline-провайдер; первые table-driven тесты на `gitlog/parser`.
- **Deliverable:** `gitl --help`, `gitl review` с флагами и двухуровневым конфигом.
- **Критерий успеха:** `go test ./...` проходит без ключа/сети; repo-level `.gitl.yaml` реально переопределяет личный конфиг с верным приоритетом.

### Неделя 3 — LLM-клиент и мультипровайдерность как контракт
- **Фокус:** надёжный HTTP-клиент + явная поддержка трёх провайдеров, а не «наверное сработает через base_url».
- **Действия:** таймауты, `context`-отмена, retry с backoff+jitter, различение ретраебельных (429/5xx) и фатальных (401/400) ошибок; интеграционные тесты/референс-конфиги для OpenAI-совместимого API, **Ollama** и **Azure OpenAI** (разные форматы auth/endpoint — вынести в `ProviderConfig`); стратегия усечения диффа (`max_diff_bytes`, `exclude_globs`).
- **Deliverable:** `internal/llm` с тестами на retry (`httptest.Server`) и тремя рабочими провайдер-конфигурациями.
- **Критерий успеха:** мок-сервер 429→200 — клиент ретраит; реальный прогон против Ollama локально и против Azure OpenAI (можно на пробном ключе) оба работают без правок кода, только конфигом.

### Неделя 4 — Риск-скоринг, версионированный JSON, cost-guard
- **Фокус:** то, что превращает «текстовое ревью» в CI-гейтящий инструмент.
- **Действия:** структурированный риск-скоринг (`low|medium|high`) как часть ответа `review` (промпт + парсинг + fallback-эвристика для offline-режима); `internal/render` с `schema_version` в JSON-выводе, задокументированная схема; `--dry-run` (оценка стоимости по токенам до вызова) и `--max-cost-usd` guard.
- **Deliverable:** `gitl review --format=json` отдаёт валидный, версионированный JSON с полем риска; `--dry-run` печатает оценку стоимости без реального вызова.
- **Критерий успеха:** `--fail-on=high` возвращает ненулевой exit code на заведомо рискованном синтетическом диффе; `--max-cost-usd=0.01` блокирует вызов с понятным сообщением на крупном диффе.

### Неделя 5 — Команды `changelog` и `digest` (в т.ч. мульти-репо)
- **Фокус:** полный набор команд.
- **Действия:** `changelog --since/--from/--to`; `digest --days=N` с агрегацией по авторам/файлам/темам; мульти-репо режим `digest --repos=a,b,c` (параллельный сбор через worker-pool — здесь естественно ложится обучение горутинам, не искусственно вынесено в опциональное «если будет настроение»); golden-тесты рендера.
- **Deliverable:** три команды, мульти-репо digest, три формата вывода.
- **Критерий успеха:** golden-тесты зелёные; digest по 3 тестовым репозиториям агрегируется корректно и не падает, если один из репо недоступен.

### Неделя 6 — GitHub Action-обёртка
- **Фокус:** CI-интеграция как часть v1, не «когда-нибудь».
- **Действия:** `action.yml` (composite или Docker action); логика постинга `review` комментарием к PR через GitHub API; выставление check status/exit code по риск-скорингу; secrets-based передача API-ключа (`secrets.GITL_API_KEY`), документация безопасного использования в CI.
- **Deliverable:** Action, прогоняемый на собственном тестовом репозитории через реальный PR.
- **Критерий успеха:** PR получает содержательный комментарий-ревью; check status отражает риск-скоринг; ключ нигде не логируется.

### Неделя 7 — Релизный инжиниринг: подпись, кросс-компиляция
- **Фокус:** supply-chain-доверие — то, без чего команды не пускают сторонний бинарь в CI.
- **Действия:** `.goreleaser.yaml` (linux/darwin/windows × amd64/arm64); `cosign`/Sigstore keyless-подписание артефактов; `SHA256SUMS`; `version` с build-provenance через `-ldflags`; semver-тег `v0.1.0`.
- **Deliverable:** подписанный GitHub Release с бинарями под 3 ОС.
- **Критерий успеха:** `cosign verify` на скачанном артефакте проходит; чек-суммы совпадают; свежескачанный бинарь запускается на чистой системе.

### Неделя 8 — Публикация Action в Marketplace, документация
- **Фокус:** товарный вид всего контура, не только CLI.
- **Действия:** публикация Action в GitHub Marketplace (иконка, описание, версионирование через теги); README (usage, BYOK/мультипровайдер-настройка, `.gitl.yaml`-политика, offline-режим, cost-guard); отдельная страница/раздел под Action-специфичную настройку (secrets, permissions).
- **Deliverable:** Action виден и устанавливаем из Marketplace; документация покрывает и CLI, и Action-сценарий.
- **Критерий успеха:** сторонний человек (или сам автор на чистом репозитории) подключает Action по README без дополнительных вопросов.

### Недели 9–10 — Хардненинг и дополнительный полиш
- **Фокус:** надёжность на реальных, а не синтетических диффах + опциональный polish.
- **Действия:** расширенный набор фикстур/edge-cases (бинарные файлы, огромные моно-диффы, ребейзы); дополнительные unit/integration тесты; по возможности — стриминг ответа в терминал, `digest --tui`, кэш ответов LLM (`~/.cache/gitl/`).
- **Deliverable:** инструмент, прогнанный на нескольких реальных репозиториях автора (dogfooding), включая Action на реальных PR.
- **Критерий успеха:** нет неожиданных падений на «грязных» реальных диффах; `-race` чист там, где есть конкурентность.

## 6) Риски и как их обойти (специфика соло-пет-проекта)

| Риск | Проявление | Обход |
|---|---|---|
| **Лимит токенов LLM на объёмном диффе** | Большой релиз не влезает в контекст → обрыв/ошибка/мусор | Порог `max_diff_bytes`; отсечение шума через `exclude_globs`; стратегия «сначала статистика (`--stat`) + сообщения коммитов, полный дифф — только для топ-N крупных файлов»; при переполнении — чанкинг по файлам с последующей агрегацией (map-reduce промпт). Реализовать усечение с явным маркером `[... diff truncated ...]`, чтобы вывод был честным. |
| **Качество AI-сводки на шумных диффах** | Сгенерированные автоформатером/локами строки забивают ревью | `exclude_globs` по умолчанию (lock/min.js/vendor/svg); низкая `temperature` (0.2); промпт с явной инструкцией игнорировать форматирование и фокусироваться на семантике; validation-first (§3.0) ловит это на неделе 1. |
| **Недетерминизм → хрупкие тесты** | LLM-вывод нельзя проверить golden-файлом | Тестировать **всё, кроме** самого вызова LLM: парсер git, рендер, retry-логику (через `httptest`), сборку промпта. LLM-слой — за интерфейсом, в тестах подменяется offline/мок-провайдером. |
| **Кросс-платформенные грабли** | Пути/конфиг ломаются на Windows | `os.UserConfigDir()`/`filepath.Join` вместо хардкода; goreleaser собирает под все ОС — ловим при сборке; ранний smoke-тест бинаря под Windows (хотя бы `--help`). |
| **Отсутствие `git` в PATH** | Паника/непонятная ошибка | Проверка наличия `git` при старте (`exec.LookPath`), дружелюбное сообщение с инструкцией. |
| **Утечка приватного диффа в LLM** | Пользователь шлёт закрытый код в чужой API | Явно задокументировать в README; поддержать локальные/enterprise провайдеры (Ollama/Azure OpenAI) для команд с требованиями резидентности данных; никакой телеметрии со стороны тула. |
| **Ложное доверие к AI-ревью при жёстком CI-гейтинге** | Команда включает `--fail-on=high`, AI ошибочно даёт high-risk на нормальном PR → блокирует мерж → разочарование → инструмент выключают | Дефолт — **WARN, не BLOCK** (жёсткий gate — явный opt-in, не поведение по умолчанию); риск-скоринг консервативно калиброван на этапе валидации (неделя 4); ревью всегда показывает reasoning, а не только вердикт — легче понять и оспорить false positive. |
| **Секреты в CI** | API-ключ утекает в логи/артефакты Action | Явная документация `secrets.*`-паттерна GitHub Actions; ключ никогда не логируется даже в verbose-режиме; тест, проверяющий отсутствие ключа в stdout/stderr при ошибках. |
| **Cost-guard не срабатывает, неожиданный счёт** | Баг в оценке стоимости → CI незаметно нагенерировал крупный LLM-счёт | `--dry-run`/`--max-cost-usd` — отдельно тестируемая логика (unit-тесты на границы), консервативный дефолт порога, явное предупреждение при отключении guard. |
| **Неподписанный артефакт не пускают в CI** | Enterprise/строгие CI-политики отклоняют исполнение неподписанных бинарей | Cosign-подпись + `SHA256SUMS` — в MVP, не post-MVP (см. §3.1); задокументированная процедура верификации в README. |
| **Выгорание интереса** | Проект брошен на середине — риск любого соло-проекта увеличенного объёма, не приукрашиваем | Не решается искусственным урезанием скоупа — решается тем же validation-first спайком (неделя 1 даёт рабочее ядро рано) и понятными еженедельными deliverable с чётким критерием «зелёно/не зелёно»; личная польза (дообился инструмент сам себе на своих репозиториях и Action) — встроенная мотивация продолжать даже при увеличенном объёме. |
| **Over-engineering ради «красоты»** | Абстракции/интерфейсы, которых никто не просил | Правило «интерфейс появляется на второй реализации, не на первой»; `pkg/` не заводить, пока не попросят внешние потребители как библиотеку. |

## 7) Бюджет — проверка по лимитам

**Разовые расходы:** практически $0.
- Компилятор Go, git, все библиотеки (cobra/viper/bubbletea/goreleaser) — бесплатны и open-source.
- GitHub (репозиторий, Releases, Actions на публичном репо, Marketplace-листинг Action) — бесплатно.
- **Cosign/Sigstore keyless signing** — бесплатен и open-source (в отличие от классических платных code-signing сертификатов).
- Домен/сайт — **не нужны** (дистрибуция через GitHub).
- Homebrew tap — бесплатный отдельный публичный репо.

**Ежемесячные расходы:** **$0/мес** инфраструктуры (нет сервера, БД, хостинга).

**Токены LLM:** платит **пользователь инструмента** своим ключом (BYOK) — разработчика это не касается. Для собственного тестирования/дев-цикла: offline-режим закрывает большую часть разработки и CI без токена; **Ollama локально** — бесплатное тестирование мультипровайдерности без внешних вызовов; тестовый прогон против **Azure OpenAI** — по пробному/минимальному кредиту (единицы $ разово, не ежемесячно). Реальные прогоны для одного разработчика на своих репозиториях — центы в месяц, с огромным запасом укладываются в лимит $50–70/мес.

**Итог:** укладывается в лимиты $500 разово / $50–70 в мес с колоссальным запасом даже с учётом добавленной продуктовой обвязки (подпись релизов, мультипровайдерное тестирование) — она стоит времени на разработку, а не денег.

## 8) Дистрибуция без активного общения

Принцип — **самообслуживание, публикуй-и-забудь**:
1. **GitHub Marketplace (Action)** — главный канал для продуктовой версии: команды находят через поиск Action по CI-задаче («AI PR review», «changelog generator»), ставят в workflow, вопросов автору не возникает.
2. **GitHub Releases** (goreleaser, подписанные cosign) — готовые бинари под 3 ОС × arm64/amd64 для CLI-сценария вне Action.
3. **`go install github.com/<user>/gitl/cmd/gitl@latest`** — для Go-аудитории нулевой порог входа.
4. **Homebrew tap** (опционально) — `brew install <user>/tap/gitl`.
5. **`awesome-go`** — один PR в список (раздел CLI/Utilities). Пассивный, долгоиграющий источник трафика.
6. **Разовый «Show HN» / пост на r/golang/r/devops** — пример вывода + ссылка на Marketplace. Ответить на первые комментарии по желанию — и всё.
7. **README + отдельная Action-документация как единственная «поддержка»** — примеры, FAQ, troubleshooting (нет ключа, 401, большой дифф, secrets в CI). Issues включены, без обязательств по SLA.

Сознательно **не делаем**: Discord/Telegram-сообщество, рассылки, регулярный контент-маркетинг, активные продажи — всё это «общение», которого разработчик избегает.

## 9) Опциональный побочный доход (по желанию, не цель)

- **Платный GitHub Marketplace Action** — раз Action теперь часть MVP (не post-MVP), это самый естественный платный тир: например, бесплатно — `review`/`changelog` с базовым риск-скорингом, платно — мульти-репо digest, кастомные политики, приоритетные модели. Биллинг целиком на стороне GitHub Marketplace — не требует своей платёжной инфраструктуры и общения с покупателями.
- **GitHub Sponsors / «Buy me a coffee»** — кнопка в README. Пассивно, ноль общения.
- **Спонсорские фичи** — «профинансируй фичу» через issues с меткой. Пассивно.
- **Реалистичная оценка:** даже с продуктовой обвязкой доход для нишевого dev-инструмента, скорее всего, скромный; это осознанно **бонус**, а не KPI. Основная «валюта» проекта — освоенные Go-навыки, портфолио и рабочий инструмент, которым автор пользуется сам.

## 10) Definition of Done

### Обучающие критерии (главное — освоенные Go-паттерны)
- [ ] Структура проекта `cmd/` + `internal/` спроектирована и обоснована; понятно, *почему* нет `pkg/`.
- [ ] cobra: дерево команд с подкомандами, персистентными и локальными флагами; viper: слияние файл/env/флаги с корректным приоритетом.
- [ ] `os/exec`: `CommandContext`, чтение stdout/stderr, коды возврата, отмена по контексту; собственный парсер git-вывода с надёжными разделителями.
- [ ] `net/http`-клиент вручную (без SDK): таймауты, `context`-отмена, retry с backoff+jitter, различение ретраебельных/фатальных ошибок через типизированные ошибки и `errors.Is`.
- [ ] (nice-to-have) горутины/каналы: worker-pool для параллельного сбора диффов, чистый `-race`.
- [ ] (nice-to-have) SSE-стриминг: разбор `data:`-чанков до `[DONE]`.
- [ ] Table-driven тесты (парсер, рендер) + golden-файлы в `testdata/` + `httptest` для LLM-клиента.
- [ ] Кросс-компиляция под 3 ОС / 2 арх через goreleaser; `version` через `-ldflags`; SemVer-тег.
- [ ] Offline-режим: полноценная разработка и CI без ключа и сети.

### Продуктовые критерии (v1 готов, CI-ready)
- [ ] Три команды (`review`, `changelog`, `digest`) работают на реальных репозиториях; `digest` — в т.ч. в мульти-репо режиме.
- [ ] `review` отдаёт машиночитаемый риск-скоринг; `--fail-on=high` корректно гейтит по exit code.
- [ ] JSON-вывод версионирован (`schema_version`) и задокументирован как контракт.
- [ ] Repo-level `.gitl.yaml` реально переопределяет личный конфиг с задокументированным приоритетом.
- [ ] `--dry-run`/`--max-cost-usd` протестированы и действительно предотвращают неограниченный расход.
- [ ] Три провайдера (OpenAI-совместимый, Ollama, Azure OpenAI) явно протестированы, не «предположительно работают».
- [ ] GitHub Action опубликован в Marketplace и прогнан на реальном PR (комментарий + check status).
- [ ] Релиз подписан cosign, `SHA256SUMS` приложены, верификация задокументирована.
- [ ] BYOK-конфиг с валидацией и дружелюбными ошибками; README покрывает и CLI, и Action-сценарий.

### Критерии внимания (сигнал, НЕ revenue)
- Разработчик **сам регулярно пользуется** `gitl` на своих репозиториях (лучший индикатор полезности).
- GitHub stars / установки / упоминания — наблюдаем пассивно; принят PR в `awesome-go`.
- Явно **не** отслеживаем: выручку, MRR, конверсии — это учебный проект, доход опционален.

### Приложение: почему именно этот проект — хороший старт портфеля
Полный жизненный цикл серьёзного, CI-годного Go-инструмента (CLI-фреймворк → внешние процессы → HTTP-клиент с мультипровайдерностью → CI-интеграция → релизный инжиниринг с подписью артефактов), а не игрушечный скрипт. Нулевая инфраструктура для автора и BYOK по-прежнему снимают расходы и биллинг — продуктовая планка поднята по качеству и governance, а не по операционной сложности/деньгам. Validation-first на неделе 1 остаётся, но не как психологическая защита от выгорания, а как стандартная инженерная дисциплина — проверить ядро, прежде чем инвестировать в CI/governance-контур вокруг него. Инструмент, которым автор пользуется сам (CLI и собственный Action на своих репозиториях), — источник мотивации и обратной связи без общения с внешними пользователями. Освоенные паттерны (`os/exec`, `net/http`+retry+мультипровайдер, горутины, goreleaser, cosign, GitHub Action-разработка) напрямую переиспользуются в более сложных последующих проектах портфеля и представляют собой более весомую строчку в портфолио, чем личный CLI-скрипт.
