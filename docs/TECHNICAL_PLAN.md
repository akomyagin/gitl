## 1. Вводные и ограничения

- Соло-разработчик, реализация с AI-ассистентом; приоритет — обучение Go
  (нагружать язык и stdlib, а не SDK).
- Целевые инфраструктурные расходы ≈ **$0/мес**; экономическая обоснованность
  необязательна.
- Требуется установленный системный `git` в `PATH` (для `os/exec`) — приемлемо,
  т.к. аудитория — разработчики.
- LLM обрабатывает **текст диффа** → жёсткий потолок по токенам → нужна стратегия
  усечения/чанкинга (см. §6, «Лимит токенов»).
- **Go 1.22+** (нужны `slices`, `maps`, `log/slog`, современный `errors.Join`).
- Кросс-платформенность: пути и конфиг через `os.UserConfigDir()`, а не хардкод.
- Без телеметрии и «звонков домой» — принципиально (доверие + приватность диффов).

## 2. Технический стек (§4.1 первоисточника)

| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | Go 1.22+ | Приоритет обучения; stdlib закрывает ~90% нужд |
| CLI | `spf13/cobra` | Стандарт индустрии (kubectl, gh, hugo), чистое дерево команд/флагов |
| Конфиг | `spf13/viper` | YAML + env + флаги в единой иерархии, интеграция с cobra |
| Git | системный `git` через `os/exec` | см. §4 |
| HTTP к LLM | `net/http` (stdlib) | Осознанно **без** SDK — тренировка ручного клиента, стриминга, retry |
| JSON | `encoding/json` (stdlib) | Парсинг ответов LLM и `--format=json` |
| TUI (opt, post-MVP) | `charmbracelet/bubbletea` + `lipgloss` | Только для `digest --tui`, поверх готовых артефактов |
| Логи | `log/slog` (stdlib) | Структурные логи, `--verbose` → debug |
| Тесты | stdlib `testing` + table-driven | Идиоматика Go; golden-файлы для рендеров |
| Сборка/релиз | `goreleaser` | Кросс-компиляция + GitHub Releases + Homebrew одним конфигом |

Сознательное ограничение зависимостей: git — stdlib `os/exec`, HTTP — stdlib
`net/http`. Внешние зависимости только там, где они дают обучающий каркас
(cobra/viper/bubbletea) или несоразмерны для ручной реализации (goreleaser).

**Почему `internal/`:** всё ядро в `internal/`, чтобы никто не импортировал
незрелый API как библиотеку — идиоматичный способ пометить «это приложение,
не библиотека». **`pkg/` намеренно не заводим на старте**: публичный API появится,
только если будет спрос («не строй библиотеку, пока её не просят»).

### 3.1 Поток данных (одна команда, §4.5)

```
cobra cmd → config.Load() → gitlog.Runner.Log/Diff(ctx, range)
   → prompt.Build(commits, diff, template)
   → llm.Client.Complete(ctx, prompt)  [retry/backoff, offline-fallback]
   → render.Render(artifact, format)   → stdout/file/TUI
```

Отмена по Ctrl-C — единый `context.Context` из `signal.NotifyContext`,
протянутый через exec и HTTP.

## 4. `os/exec` vs `go-git` — решение (§4.3)

**Выбор: системный `git` через `os/exec`.** Причины, релевантные цели
«написание + обучение Go»:

1. **Тренирует ядровой Go-навык** — работу с внешними процессами:
   `exec.CommandContext`, чтение `stdout`/`stderr` через `bytes.Buffer`/
   `bufio.Scanner`, коды возврата, контекст-отмена.
2. **Заставляет писать парсер** — самая обучающая часть: спроектировать
   `--pretty=format:` с разделителями, распарсить `--name-status`, покрыть
   table-driven-тестами на golden-выводе. С `go-git` данные приходят структурами —
   упражнение исчезает.
3. **Простая ментальная модель** — поведение 1:1 с терминалом; отлаживается
   тривиально (скопировал команду, запустил вручную).
4. **`go-git` тяжелее и с нюансами** — не 100% паритет с CLI по краевым случаям,
   больше зависимость.

Компромисс инкапсулируется: весь `os/exec` спрятан за интерфейсом в
`internal/gitlog` — если позже захочется `go-git`, меняется одна реализация,
команды не трогаются.

**Парсинг git log с control-разделителями (критичная деталь):**

```
git log --pretty=format:%H%x1f%an%x1f%aI%x1f%s%x1f%b%x1e --name-status <range>
```

- `%x1f` (ASCII **unit separator**, 0x1F) разделяет **поля** внутри коммита.
- `%x1e` (ASCII **record separator**, 0x1E) разделяет **коммиты**.

Control-символы из ASCII-зоны 0x1F/0x1E не встречаются в сообщениях коммитов,
поэтому надёжны как разделители. **Частая ошибка новичка — split по `\n`**: тело
коммита содержит и переводы строк, и пробелы, поэтому наивный split ломается.

Целевые типы (`internal/gitlog/types.go`):

```go
type Commit struct {
    Hash    string
    Author  string
    Date    time.Time
    Subject string
    Body    string
    Files   []FileChange
}
type FileChange struct {
    Status string // A, M, D, R...
    Path   string
    Old    string // для переименований (R)
}
```

## 5. Схема конфигурации (§4.4)

Два уровня, сливаются по приоритету
**флаг > env > `.gitl.yaml` (репозиторий) > `~/.config/gitl/config.yaml` (личный)**.
Личный файл — путь через `os.UserConfigDir()` (на Windows → `%AppData%\gitl`);
repo-level `.gitl.yaml` в корне репозитория коммитится в git и является общей
политикой команды.

```yaml
# ~/.config/gitl/config.yaml — личные настройки (секреты, провайдер по умолчанию)
llm:
  provider: "openai"          # "openai" | "ollama" | "azure_openai" — явный контракт
  api_key: ""                 # или env GITL_API_KEY (рекомендовано, особенно в CI)
  base_url: "https://api.openai.com/v1"
  model: "gpt-4o-mini"
  max_tokens: 1500
  temperature: 0.2
  timeout_seconds: 60
  max_retries: 3
  # azure_openai: endpoint, deployment, api_version — под-блок при provider=azure_openai
  # ollama: host (по умолчанию http://localhost:11434) — без api_key

cost:
  max_cost_usd: 0.50          # cost-guard: отказ выполнять запрос дороже порога
  warn_at_usd: 0.10           # предупреждение, но не блокировка

output:
  format: "md"                # md | text | json
  color: true

diff:
  max_diff_bytes: 120000
  exclude_globs:
    - "*.lock"
    - "*.min.js"
    - "vendor/**"
    - "*.svg"
```

```yaml
# .gitl.yaml — repo-level политика (коммитится в репозиторий, общая для команды)
policy:
  fail_on: "high"             # low | medium | high | never — порог гейтинга CI
  required_changelog_categories: ["Added", "Fixed", "Security"]
  exclude_globs: ["migrations/**"]   # ДОБАВЛЯЕТСЯ к личным exclude_globs, не заменяет

digest:
  repos:                      # мульти-репо digest
    - path: "../service-a"
    - path: "../service-b"
```

**Валидация при старте:** если `api_key` пуст/недоступен → offline-режим
с предупреждением в stderr (не падаем). Провайдер выбирается явным полем
`provider`, а не угадыванием по `base_url` — это и есть контракт
мультипровайдерности.

## 7. Контракт риск-скоринга и версионированного JSON-вывода (детали Этапа 2)

> Дизайн-решения, зафиксированные **до** реализации Этапа 2, чтобы не изобретать
> схему по ходу кодирования. Дополняет §6 «Этап 2».

### 7.1 Как модель сообщает риск

Однократный вызов LLM, без второго round-trip ради скоринга (дороже и сложнее
retry-логики). `reviewSystem`-промпт (`internal/prompt/templates.go`) получает
четвёртый обязательный пункт вдобавок к Summary/Notable changes/Concerns:

> 4. Закончи ответ отдельным fenced-блоком с языком `risk`, содержащим ровно
> один однострочный JSON-объект, и это должно быть последним, что ты выводишь:
> `{"level": "low|medium|high", "summary": "<одно предложение, до 140 символов>"}`.

Пример хвоста ответа модели:

````
## Concerns

- ...

```risk
{"level": "medium", "summary": "Touches auth middleware without new tests."}
```
````

### 7.2 Парсинг и fallback

- Ищем **последний** fenced-блок с языком `risk` (терпимо принимаем и ```json,
  если внутри валидный объект с ключом `level`); `level` приводим к нижнему
  регистру и валидируем как `low|medium|high`.
- Валидный блок вырезается из текста перед рендером — пользователь не должен
  видеть сырой JSON в Markdown/text-выводе.
- Если блока нет / JSON невалиден / `level` не из трёх значений — эвристический
  fallback (§7.3), с `slog.Warn` о том, что оценка модели не распарсилась.
- **Offline-провайдер всегда** использует эвристику (нет модели, которая могла
  бы дать риск) — он и раньше это делал по духу, теперь формально возвращает
  структурированное поле, а не только текст.

### 7.3 Эвристика (offline-режим и fallback при сбое парсинга)

Одна детерминированная функция на оба случая (одинаковый вход → одинаковый
выход, без сети и случайности — правило офлайн-провайдера из SKILL.md):

- **high**, если diff задевает чувствительные пути/ключевые слова
  (`auth`, `secret`, `password`, `token`, `credential`, `security`, `payment`,
  `migration`, `permission` — регистронезависимый поиск по путям файлов,
  темам и телам коммитов) **или** изменённых строк больше ~300, **или**
  затронуто больше ~20 файлов.
- **low**, если изменённых строк меньше ~20, файлов ≤ 3 и чувствительных
  совпадений нет.
- **medium** — всё остальное (дефолт).
- `summary` — короткое шаблонное объяснение сработавшего правила (например,
  `"Large diff (312 changed lines) across 14 files."` или
  `"Touches security-sensitive keywords: \"auth\", \"token\"."`).

Точные пороги — не догма, их можно откалибровать при реализации; важна
детерминированность и то, что offline и online-fallback используют одну
и ту же функцию (не две параллельные реализации).

### 7.4 JSON-схема вывода (`--format=json`, `schema_version: 1`)

```json
{
  "schema_version": 1,
  "generated_at": "2026-07-05T18:32:10Z",
  "range": "HEAD~5..HEAD",
  "offline": false,
  "provider": "openai",
  "model": "gpt-4o-mini",
  "risk": {
    "level": "medium",
    "summary": "Touches auth middleware without new tests."
  },
  "stats": {
    "commits": 5,
    "files_changed": 12,
    "lines_added": 340,
    "lines_removed": 58
  },
  "commits": [
    {
      "hash": "abcd123",
      "author": "Jane Doe",
      "date": "2026-07-01T10:00:00Z",
      "subject": "feat: add token refresh"
    }
  ],
  "review_markdown": "## Summary\n...\n## Notable changes\n...\n## Concerns\n..."
}
```

Правила версии: `schema_version` — целое число, бампается только при breaking
change (переименование/удаление поля, смена типа); добавление нового
необязательного поля версию не меняет. `review_markdown` — тот же текст, что
идёт в `--format=md` (без хвостового `risk`-блока, он уже вынесен в `risk`);
`--format=text` — тот же контент с грубо срезанной Markdown-разметкой
(`#`/`**`/бэктики), без отдельного JSON-парсинга. `risk` всегда заполнено
(offline и fallback гарантируют значение) — консьюмеру в CI не нужно
обрабатывать `null`.

### 7.5 Точки изменения в уже написанном коде Этапа 1 (ориентир)

- `llm.Request` получает `Commits []gitlog.Commit` и `Diff string` —
  pass-through, чтобы эвристика (§7.3) считалась внутри провайдера без
  повторного парсинга текста промпта; `llm.Response` получает поле `Risk`.
- Общая `llm.HeuristicRisk(commits, diff) Risk` — вызывается и из `Offline`,
  и как fallback из `Client` при нераспарсенном риск-блоке.
- `render.Render` меняет вход с голой строки на артефакт, несущий поля §7.4
  (range/risk/stats/commits/offline/provider/model/review_markdown); md-рендер
  добавляет короткую шапку `**Risk:** MEDIUM — summary` перед текстом ревью.
  Конкретные имена Go-типов — на усмотрение реализации, wire-контракт JSON
  выше — обязателен.
- Таблица цен для `--dry-run`/`--max-cost-usd` в этот дизайн не входит —
  отдельное решение перед стартом Этапа 2.

## 8. Оценка стоимости и cost-guard (детали Этапа 2)

> Дизайн `--dry-run`/`--max-cost-usd`, зафиксированный до реализации. Дополняет
> §6 «Этап 2». Без реального токенайзера и без вызова стороннего pricing-API —
> в духе «$0/мес инфраструктуры» и офлайн-совместимости.

### 8.1 Оценка токенов

Грубая эвристика, не точный расчёт (не `tiktoken`): `EstimateTokens(text) =
ceil(len(text_in_bytes) / 4)` — общепринятое приближение «~4 символа на
токен». Считается по итоговому промпту (system+user), который реально уйдёт
в API. Выходные токены не гадаем по содержанию — берём сконфигурированный
`llm.max_tokens` как потолок (это и есть то число, что уходит в запрос).
Итоговая оценка токенов умножается на **×1.15** — сознательный запас в сторону
переоценки: ошибка "посчитали дороже, чем есть" безопасна (просто просит
поднять лимит), ошибка "посчитали дешевле" — именно то, от чего guard должен
защищать.

### 8.2 Таблица цен + override

Встроенная таблица (например, `internal/llm/cost.go`), цены **за 1M токенов**,
ориентировочные на момент написания плана (2026-07, сверено с
developers.openai.com/api/docs/pricing и сторонними агрегаторами) —
устаревание ожидаемо и не является багом, но стоит перепроверить перед
реализацией:

| Модель | Input / 1M | Output / 1M |
|---|---|---|
| `gpt-4o-mini` (дефолт `llm.model`) | $0.15 | $0.60 |
| `gpt-4o` | $2.50 | $10.00 |
| `gpt-4.1-mini` | $0.40 | $1.60 |

- **Explicit override** в конфиге: `cost.price_per_1m_input` /
  `cost.price_per_1m_output` (float64, `0` = использовать встроенную таблицу).
  Перекрывает таблицу безусловно — нужен для моделей не из таблицы,
  устаревших цен без правки кода, self-hosted OpenAI-совместимых эндпоинтов
  с другим прайсингом.
- **`provider: ollama`** — стоимость всегда `$0` (self-hosted); guard/оценка
  пропускаются целиком, без предупреждений (ожидаемый бесплатный путь).
- **Модель не найдена в таблице и override не задан** (провайдер не ollama) —
  guard **не блокирует**: явное предупреждение в stderr
  (`no pricing data for model %q — cost estimate unavailable, proceeding
  without cost-guard (set cost.price_per_1m_input/output to enable it)`) и
  выполнение продолжается без проверки. Permissive-by-default — кастомная
  модель не должна молча падать из-за отсутствия в таблице.

### 8.3 `--dry-run`

- Никакого сетевого вызова. Печатает в **stdout** (это полезная нагрузка
  команды, не диагностика): оценку input-токенов, потолок output-токенов,
  итоговую оценку в USD, модель/провайдера и явную пометку «estimate, not
  exact» — и завершает работу с exit code 0, не вызывая провайдера.
- В offline-режиме (`cfg.OfflineMode()`) `--dry-run` печатает «offline mode —
  no API call, no cost» и выходит 0, а не падает с ошибкой.

### 8.4 `--max-cost-usd` guard

- Считается тем же путём, что и `--dry-run`, но встроен в обычный `review`
  (не отдельная команда), выполняется **до** вызова провайдера, сразу после
  сборки промпта.
- `cost.max_cost_usd` из конфига, переопределяется флагом `--max-cost-usd`
  (через уже существующий механизм `bindChangedFlags`).
- `estimate > max_cost_usd` → ошибка, ненулевой exit code, сообщение с
  цифрами и путями решения, например:
  `gitl: estimated cost $0.42 (≈14,231 input + 1,500 output tokens @
  gpt-4o-mini) exceeds --max-cost-usd=0.01 — increase the limit, use
  --dry-run to inspect, or omit the API key for a free offline review.`
- `warn_at_usd < estimate ≤ max_cost_usd` → предупреждение в stderr,
  выполнение продолжается.
- `max_cost_usd <= 0` — guard явно отключён («без лимита»); это осознанный
  выбор пользователя (документируется в README), не дефолт (дефолт остаётся
  `$0.50` из конфига Этапа 1).
- Guard полностью пропускается в offline-режиме — там нет вызова и нет
  стоимости.

### 8.5 Точки изменения в коде (ориентир)

- Новый `internal/llm/cost.go`: `EstimateTokens`, встроенная таблица цен,
  `EstimateCost(...) (Estimate, error)` — чистые, детерминированные функции;
  юнит-тесты на границы, включая тест, прямо воспроизводящий Критерий успеха
  Этапа 2: `--max-cost-usd=0.01` блокирует крупный синтетический дифф.
- `config.CostConfig` получает `PricePer1MInput`/`PricePer1MOutput`
  (`mapstructure: price_per_1m_input`/`price_per_1m_output`).
- `cli/review.go`: флаги `--dry-run` (bool) и `--max-cost-usd` (float64,
  добавлен в `bindChangedFlags` → `cost.max_cost_usd`); guard вызывается
  после сборки промпта (нужен реальный текст запроса), до `selectProvider`/
  `provider.Complete`.

## 9. Контракт `changelog` и `digest` (детали Этапа 3)

> Дизайн-решения, зафиксированные **до** реализации Этапа 3, тем же способом,
> каким §7/§8 зафиксировали риск-скоринг и cost-guard перед Этапом 2. Дополняет
> §6 «Этап 3». Три новые команды используют **уже существующий** `gitlog.Runner`
> и `internal/render` — новых путей похода в git или новых пакетов рендера
> не заводим без необходимости (правило «интерфейс на второй реализации», не
> «пакет про запас»).

### 9.1 `changelog` — флаги и маппинг на `git log`

**Флаг: `--range` (не `--since`/`--from`/`--to`).** `review` уже принимает
диапазон как позиционный аргумент (`review <range>`, например `HEAD~5..HEAD`).
Ради консистентности `changelog` тоже принимает **позиционный `<range>`**
(тот же формат `A..B`, включая голый `HEAD`/тег/SHA — всё, что понимает
`git log`), а не отдельные `--since`/`--from`/`--to` флаги:

```
gitl changelog [<range>] [--format=md|text|json]
```

- `<range>` — **необязательный** позиционный аргумент. Если не передан,
  по умолчанию — **диапазон от последнего annotated/lightweight тега до
  `HEAD`** (`git describe --tags --abbrev=0` → `<tag>..HEAD`); если тегов
  в репозитории нет вовсе — весь `HEAD` (эквивалент `git log HEAD`, т.е.
  вся история). Это единственное «умное» поведение по умолчанию во всём
  Этапе 3 — обоснование: changelog почти всегда пишут «с прошлого релиза»,
  и заставлять пользователя каждый раз искать тег вручную — потерянная
  эргономика при нулевой цене реализации (`exec.LookPath` уже есть,
  добавляется один `git describe` вызов).
- Ошибка `git describe` (нет тегов) — не фатальна, просто откат на весь
  `HEAD` с `slog.Debug` (не `Warn`: отсутствие тегов в раннем репозитории —
  норма, не проблема конфигурации).
- Никакого отдельного `--since`/`--from`/`--to` не вводим: это дублировало бы
  семантику `<range>`, которую `git log` и так понимает через revision syntax
  (`v1.0.0..HEAD`, `HEAD~10..`, `2026-01-01..HEAD` через `@{<date>}` и т.п.
  пользователь передаёт range-выражение напрямую, gitl его не парсит и не
  переинтерпретирует).
- Реализация похода в git: **переиспользуется `gitlog.Runner.Log(ctx, range)`**
  — тот же вызов, что и в `review`. Второй `git log` (для авто-определения тега)
  — новый маленький метод на `Runner` (см. §9.5), а не второй независимый
  `exec.Command` где-то в `cli/changelog.go`: весь `os/exec` остаётся
  инкапсулирован в `internal/gitlog` (правило пакетных границ, §3).
- `changelog` **не вызывает** `git diff` и **не вызывает LLM** — категоризация
  полностью детерминирована по метаданным коммитов (см. §9.2). Обоснование —
  то же, что для `digest` (§9.6): Keep a Changelog группирует по типу
  изменения, который уже явно закодирован в conventional-commit префиксе;
  прогонять через LLM то, что и так однозначно парсится, — трата токенов
  и источник недетерминизма там, где он не нужен. Это отличает `changelog`
  от `review`, для которого LLM — основная ценность.

### 9.2 Категоризация conventional-commits → Keep a Changelog

Категория определяется **только по префиксу `Subject`** (не по `Body`,
не по файлам) — предсказуемо и соответствует тому, как разработчики реально
пишут conventional-commits. Точный маппинг префикс → категория:

| Префикс (регистронезависимо, до `:`) | Категория Keep a Changelog |
|---|---|
| `feat` | Added |
| `fix` | Fixed |
| `perf`, `refactor` | Changed |
| `revert` | Changed |
| `deprecate` (нестандартный, но однозначный) | Deprecated |
| `remove` (нестандартный, но однозначный) | Removed |
| `security` (нестандартный, но однозначный) | Security |
| `docs`, `style`, `test`, `build`, `ci`, `chore` | **Other** |
| что угодно ещё / нет `type:`-префикса вовсе | **Other** |

- Парсинг префикса: `^(\w+)(\([^)]*\))?(!)?:\s`, т.е. `type`, опциональный
  `(scope)`, опциональный breaking-маркер `!`, затем `:` и пробел
  (стандартная conventional-commits грамматика). Совпадение по `type` —
  регистронезависимое, но в выводе категория печатается в каноническом
  Keep a Changelog регистре (`Added`, не `added`).
- **Breaking change:** коммит считается breaking, если **либо** `type!:`
  (маркер `!` сразу после типа/скоупа), **либо** `Body` содержит строку,
  начинающуюся с `BREAKING CHANGE:` (стандартный footer conventional-commits,
  поиск по началу строки в `Body`, регистрозависимо — это фиксированная
  условность спецификации). Breaking-коммит **не** заводит отдельную
  Keep a Changelog категорию (в спеке Keep a Changelog такой категории нет);
  вместо этого:
  - категория назначается как обычно по типу (например, `feat!:` → всё ещё
    `Added`, `fix!:` → `Fixed`), **и**
  - запись в этой категории получает суффикс `` **BREAKING**: `` перед
    текстом (см. пример рендера в §9.4), и коммит дополнительно попадает
    в отдельную секцию `## ⚠ BREAKING CHANGES` **в начале** документа
    (перед `Added`/`Changed`/...), с текстом — первой строкой после
    `BREAKING CHANGE:` в `Body`, либо, если маркер был только `!` без
    футера, самим `Subject` без префикса.
- **Non-conventional коммиты (нет `type:` вовсе)** — попадают в **Other**,
  не отбрасываются и не варнятся. Обоснование: история реальных репозиториев
  всегда содержит смешанные коммиты (мержи, ручные фиксы без конвенции,
  коммиты до принятия конвенции в проекте) — молчаливый дроп терял бы
  changelog-видимые изменения, а warning на каждый такой коммит в типичном
  репозитории создаст шум, который научит игнорировать warnings вообще.
  Merge-коммиты (`Subject` начинается с `Merge `) — единственное исключение:
  они **исключаются** из всех категорий целиком (ни в Other, ни в счётчиках),
  потому что несут нулевую changelog-информацию сами по себе (изменения уже
  представлены их собственными коммитами в диапазоне).
- Дедупликация: как и в offline-провайдере ревью, если один и тот же `Hash`
  почему-то встретится дважды (не должно происходить при нормальном
  использовании `git log`, но защита дешёвая) — оставляем первое вхождение.

### 9.3 `policy.required_changelog_categories` — конкретное поведение

Поле уже есть в схеме конфига (`PolicyConfig.RequiredChangelogCategories`,
`.gitl.yaml` пример в §5), но не подключено никуда. Этап 3 подключает его
с точным поведением: **warn, не fail**.

- После категоризации: для каждой строки в `policy.required_changelog_categories`
  (например `["Added", "Fixed", "Security"]`) проверяется, есть ли в получившемся
  changelog хотя бы одна запись в этой категории.
- Если категория из списка **пуста** — `slog.Warn` в stderr вида:
  `required changelog category "Security" has no entries in range %q`
  — **один warning на каждую пустую required-категорию**, не одна общая
  строка. Changelog всё равно печатается и exit code остаётся `0`.
- **Никакого `--fail-on` для changelog в Этапе 3.** Обоснование: в отличие
  от `review`, где риск-скоринг — это оценка опасности, пустая категория
  changelog — это не риск, а зачастую нормальная ситуация (например, в этом
  диапазоне действительно не было security-фиксов). Жёсткий gate здесь скорее
  наказывал бы за отсутствие бага, чем ловил реальную проблему; это
  сознательно другая семантика от risk-скоринга `review`, и смешивать их
  в одном `policy.fail_on` было бы категорийной ошибкой. Если в будущем
  появится спрос на жёсткий gate по required-категориям — это отдельное,
  явно новое решение post-MVP, не задел здесь.
- Список пустой/не задан (`nil`/`[]`) — проверка тривиально пропускается,
  без каких-либо сообщений.

### 9.4 Формат вывода `changelog`

**Markdown** — канонический Keep a Changelog вид:

```markdown
## [Unreleased] — v1.2.0..HEAD

### ⚠ BREAKING CHANGES

- **BREAKING**: drop support for config schema v0 (auth.go)

### Added

- add token refresh endpoint (a1b2c3d)
- **BREAKING**: rework session store API (d4e5f6a)

### Fixed

- correct off-by-one in pagination (9f8e7d6)

### Other

- update contributing guide (1122334)
```

- Заголовок диапазона `## [Unreleased] — <range>` — фиксированная строка
  `[Unreleased]`, так как gitl не знает целевую версию релиза (это решает
  человек/CI при тегировании, не входит в скоуп Этапа 3 — нет `--version`
  флага, версионирование релиза не входит в это issue). `<range>` — то же
  значение, что и разрешённый `<range>` из §9.1 (после авто-определения тега).
- Категории печатаются **только если непустые**, в фиксированном порядке:
  `⚠ BREAKING CHANGES, Added, Changed, Deprecated, Removed, Fixed, Security, Other`
  (порядок Keep a Changelog + `BREAKING`/`Other` на своих местах).
- Каждая запись — `<subject без префикса type(scope)!:> (<короткий hash>)`;
  `<короткий hash>` — 7 символов, как в offline-провайдере ревью (единый
  стиль по всему проекту).
- **`--format=text`** — тот же контент через уже существующий
  `stripMarkdown` (переиспользуется как есть, без изменений: он оперирует
  над произвольной Markdown-строкой, ничего специфичного для review в нём нет).

**JSON-схема (`schema_version`):** **отдельное схема-пространство от
`review`**, не тот же счётчик версий. Обоснование: `review`'s
`schema_version: 1` — это версия конкретного wire-контракта одной команды
(`risk`/`stats`/`commits`/`review_markdown`); `changelog` — структурно
не связанный контракт с другим потребителем (CI-парсер changelog, не
риск-гейтинг). Общий числовой namespace означал бы, что breaking change
в JSON `digest` бампает номер, который читает консьюмер `review --format=json`,
не тронув ни одного его поля, — ложная сигнализация. Вместо этого каждая
команда версionирует свой JSON независимо, но **одним и тем же именем поля**
`schema_version` (контракт есть contract per-command, имя поля унифицировано
ради единообразия для тех, кто парсит вывод нескольких команд):

```json
{
  "schema_version": 1,
  "command": "changelog",
  "generated_at": "2026-07-06T09:00:00Z",
  "range": "v1.2.0..HEAD",
  "breaking_changes": [
    {"hash": "d4e5f6a", "subject": "drop support for config schema v0", "category": "Added"}
  ],
  "categories": {
    "Added": [
      {"hash": "a1b2c3d", "subject": "add token refresh endpoint", "breaking": false},
      {"hash": "d4e5f6a", "subject": "rework session store API", "breaking": true}
    ],
    "Changed": [],
    "Deprecated": [],
    "Removed": [],
    "Fixed": [
      {"hash": "9f8e7d6", "subject": "correct off-by-one in pagination", "breaking": false}
    ],
    "Security": [],
    "Other": [
      {"hash": "1122334", "subject": "update contributing guide", "breaking": false}
    ]
  },
  "missing_required_categories": ["Security"]
}
```

- `command` — новое поле, отсутствующее в `review`'s схеме; явная метка
  для консьюмеров, которые видят JSON нескольких gitl-команд вперемешку
  (например, в одном CI-пайплайне).
- `categories` — объект с **всеми семью** ключами Keep a Changelog всегда
  присутствующими (даже с пустым массивом `[]`), **кроме** `Changed`/
  `Deprecated`/`Removed`/`Security`, которые в примере выше пусты, но
  ключ есть — это чтобы консьюмер мог написать `categories.Security.length`
  без проверки на `undefined`. `⚠ BREAKING CHANGES` в JSON — не категория
  Keep a Changelog, а отдельный массив верхнего уровня `breaking_changes`
  (категория в каждой записи там присутствует — то, в какую Keep a Changelog
  секцию попал этот же коммит).
- `missing_required_categories` — прямое отражение проверки §9.3, всегда
  массив (пустой, если всё покрыто или список required пуст) — CI может
  само решить, фейлиться на непустом массиве или только логировать, раз
  gitl сам не фейлит.
- `hash` — короткий (7 символов), как и в Markdown-выводе — единообразие
  между форматами одной команды.

### 9.5 Точки изменения в `internal/gitlog` для `changelog`

- Новый метод на `Runner`/`Source`: `LatestTag(ctx context.Context) (string, error)`
  — обёртка над `git describe --tags --abbrev=0`; при ошибке (нет тегов)
  возвращает `("", nil)` — **не ошибку** (отсутствие тегов — легитимный
  результат, не сбой git), чтобы вызывающий код в `cli/changelog.go` мог
  просто проверить `tag == ""` без развилки по типу ошибки. `Source`
  расширяется этим методом (второй метод интерфейса, не новый интерфейс —
  `Source` уже существует и уже абстрагирует `os/exec`).
- Никаких изменений в `ParseLog`/`types.go` не требуется — `changelog`
  использует уже существующие `Commit.Subject`/`Body`/`Hash`.
- Новый файл `internal/gitlog/changelog.go` (не `cli/changelog.go`) для чистых
  функций категоризации:
  `CategorizeCommits(commits []Commit) Changelog` (или аналогичное имя),
  где `Changelog` — новый тип, лежащий рядом с `Commit`/`FileChange`
  в `internal/gitlog` (не в `internal/render` — категоризация это чтение
  git-данных, семантически принадлежит `gitlog`, а не рендерингу; `render`
  получает уже готовую структуру и только форматирует её, как и для `review`).
  Table-driven тесты категоризации (§9.2 маппинг, breaking-детекция,
  merge-исключение, дедуп) живут в `internal/gitlog/changelog_test.go`.

### 9.6 `internal/render` — расширение для `changelog`

Один пакет `render`, новый артефакт-тип и новые render-функции — **не**
новый пакет. Обоснование: `render` уже параметризован по `Format` и уже
инкапсулирует md/text/json-механику (`stripMarkdown`, JSON-энкодинг
с отступом, риск-заголовок); дублировать это в `internal/render/changelog.go`
как отдельный пакет было бы преждевременной изоляцией ради несуществующей
проблемы. Конкретно:

- Новый файл `internal/render/changelog.go` в том же пакете `render`.
- Новый тип `ChangelogArtifact` (аналог `Artifact` для review) со своим
  `ChangelogSchemaVersion = 1` — **отдельная константа**, не переиспользование
  `SchemaVersion` (см. обоснование раздельного версионирования в §9.4).
- Новая функция `RenderChangelog(w io.Writer, art ChangelogArtifact, format Format) error`
  — не перегрузка существующего `Render` (тот принимает `Artifact`
  конкретного review-типа; заводить `any`/интерфейс ради двух вызывающих
  было бы преждевременной абстракцией — «интерфейс на второй реализации»
  здесь трактуется так: у нас будет **вторая функция**, не вторая реализация
  одного интерфейса, а значит общий интерфейс `Render`/`RenderChangelog`
  не нужен вовсе, пока не появится третий формат-агностичный потребитель).
  `cli/changelog.go` вызывает `render.RenderChangelog(...)` напрямую.
- `renderText` для changelog переиспользует существующий пакетный
  `stripMarkdown` (уже общая функция, не привязанная к `Artifact`).

## 10. Контракт `digest` и мульти-репо worker-pool (детали Этапа 3)

> Дополняет §6 «Этап 3» и §9 выше. `digest` — по духу «offline-first» команда:
> она **не вызывает LLM вообще** (см. обоснование в §10.2) и целиком
> детерминирована по git-метаданным, поэтому не имеет offline/online режима
> в смысле `review` — она всегда работает одинаково, с ключом или без.

### 10.1 `digest --days=N` — семантика окна (однорепо)

```
gitl digest [--days=N] [--repos=a,b,c] [--format=md|text|json]
```

- `--days` — `int`, по умолчанию **7** (задокументированный дефолт, как
  и остальные числовые пороги в проекте). `--days=0` или отрицательное —
  ошибка валидации флага (`--days must be a positive integer, got %d`),
  а не молчаливый фоллбэк — в отличие от `cost.max_cost_usd<=0`, у которого
  «0 = выключено» — это осмысленный семантический режим; для `--days` такого
  режима нет («digest без окна» не имеет смысла, а `--days=0` typo вместо
  `--days=30` не должен молча дать пустой digest).
- Окно вычисляется от **текущего момента** (`time.Now().UTC()`), не от
  `HEAD` коммита — команда называется «за последние N дней» в календарном
  смысле, а не «последние N дней активности до последнего коммита» (эти
  два расходятся, если в репозитории давно не коммитили — второе дало бы
  сюрприз «почему digest за 7 дней показывает коммит годичной давности»).
- Git-вызов: **новый метод** `Runner.LogSince(ctx, since time.Time) ([]Commit, error)`,
  реализованный как `git log --since=<RFC3339> --pretty=format:...`
  (тот же `logFormat`+`--name-status`, что и `Log`, только вместо
  позиционного `<range>` — флаг `--since`; `<range>` и `--since` в `git log`
  комбинируются, но здесь используется только `--since` без range-аргумента,
  т.е. неявно «от `--since` до `HEAD`» — ветка по умолчанию текущего чекаута,
  ровно то же ветко-поведение, что и остальные команды). `since` форматируется
  как `time.Now().UTC().AddDate(0, 0, -days).Format(time.RFC3339)` — RFC3339,
  который `git log --since` понимает нативно, без специальной парсер-логики
  на стороне gitl.
- **Пустой результат (0 коммитов за N дней)** — не ошибка: digest печатается
  с нулевой статистикой и пустыми секциями агрегации (аналогично тому, как
  `review` требует ≥1 коммит и фейлится — но `digest` **сознательно другая
  политика**: обзор активности за период, где «активности не было», это
  валидный и полезный ответ, а не пользовательская ошибка диапазона).

### 10.2 Агрегация: авторы / файлы / темы — точные алгоритмы

Все три агрегации — чистые функции над `[]gitlog.Commit`, детерминированные,
без сети и LLM. **`digest` никогда не вызывает LLM** — ни online, ни offline-
эвристику `llm.HeuristicRisk` (там нет риск-скоринга вообще, это не review).
Обоснование зафиксировано явно: агрегация по автору/файлу — это подсчёт,
не оценка; аггрегация «по теме» (см. ниже) выводится из той же
conventional-commit категоризации, что уже разработана для `changelog`
(§9.2) — переиспользование, а не второй параллельный классификатор. Раз
вся нужная информация извлекается детерминированно из существующих полей
`Commit`, вызов LLM был бы вводимой недетерминированностью без выигрыша
в качестве — противоречит принципу «offline-first» проекта (PLAN.md §2).

- **По авторам** (`AuthorStats`): группировка коммитов по `Commit.Author`
  (точное строковое совпадение — normalize/mailmap не входит в скоуп,
  post-MVP-кандидат при появлении спроса), для каждого — счётчик коммитов
  и сумма `+`/`-` строк по `gitlog.DiffLineStats` **дифф-текста только этого
  коммита**. Дифф на коммит нужен отдельно (не общий диапазонный дифф,
  как в `review`) — новый метод `Runner.DiffForCommit(ctx, hash) (string, error)`
  (обёртка над `git show --format= <hash>`, т.е. дифф ровно одного коммита
  без сообщения). Список авторов сортируется по числу коммитов **убыв.**,
  при равенстве — по имени **возр.** (детерминированный tie-break, важно
  для golden-тестов).
- **По файлам** (`FileStats`): подсчёт числа затрагивающих коммитов на
  каждый уникальный `FileChange.Path` (не строк, а именно коммитов —
  «сколько раз этот файл менялся», не построчная churn-метрика — она уже
  выражена в общей статистике; смешивать две единицы измерения в одной
  таблице запутывает). Топ-10 по убыванию числа коммитов, tie-break —
  по пути **возр.** (лексикографически). Ограничение топ-10 — чтобы
  digest не превращался в полный листинг файлов монорепо; число `10`
  — захардкожено (не конфигурируемо в Этапе 3 — сознательно, чтобы не
  плодить флаги под то, что не подтверждено спросом).
- **По темам** (`TopicStats`) — конкретное, детерминированное определение:
  тема = **conventional-commit type** коммита, определённый той же функцией
  парсинга префикса, что и в §9.2 (`feat`, `fix`, `refactor`, ...,
  нестандартные/безпрефиксные — тема `"other"`). Это единственная трактовка
  «темы», совместимая с «offline-first, детерминированный» принципом:
  альтернатива (кластеризация по NLP/embeddings над сообщениями коммитов)
  требовала бы LLM или векторной модели — прямо то, от чего `digest`
  сознательно отказывается (см. выше). Вывод: подсчёт коммитов на каждый
  conventional-commit type, отсортированный по убыванию числа коммитов,
  tie-break — по имени типа **возр.**

### 10.3 Формат вывода `digest` (однорепо)

**Markdown:**

```markdown
# Digest — last 7 days (2026-06-29 → 2026-07-06)

## Summary

- Commits: 23
- Files touched: 14
- Lines: +812 / -145

## By author

| Author    | Commits | Lines (+/-) |
|-----------|---------|-------------|
| Jane Doe  | 14      | +610 / -90  |
| John Roe  | 9       | +202 / -55  |

## By topic

| Topic (type) | Commits |
|--------------|---------|
| feat         | 10      |
| fix          | 7       |
| other        | 6       |

## Top changed files

| File               | Commits |
|--------------------|---------|
| internal/llm/client.go | 5   |
| README.md              | 3   |
```

- Заголовок диапазона печатает обе границы окна в `YYYY-MM-DD` (человеко-
  читаемо), несмотря на то что фактический git-вызов использует RFC3339 —
  разные представления для разных целей (машинный вызов vs человеческий
  вывод), не два источника правды (оба выводятся из одного `since`/`now`).
- `--format=text` — тот же `stripMarkdown`, таблицы вырождаются в
  плоский `Author | N | +X/-Y` построчно (markdown-таблицы и так не
  рендерятся красиво без `#`/`**`/backtick — `stripMarkdown` их не трогает
  специально, что приемлемо: `text` уже документирован как «грубый», см. §7.4).

**JSON (`schema_version`, отдельный namespace от `review` и `changelog` — то
же обоснование §9.4):**

```json
{
  "schema_version": 1,
  "command": "digest",
  "generated_at": "2026-07-06T09:00:00Z",
  "days": 7,
  "since": "2026-06-29T09:00:00Z",
  "until": "2026-07-06T09:00:00Z",
  "stats": {
    "commits": 23,
    "files_changed": 14,
    "lines_added": 812,
    "lines_removed": 145
  },
  "by_author": [
    {"author": "Jane Doe", "commits": 14, "lines_added": 610, "lines_removed": 90}
  ],
  "by_topic": [
    {"topic": "feat", "commits": 10}
  ],
  "top_files": [
    {"path": "internal/llm/client.go", "commits": 5}
  ]
}
```

### 10.4 Мульти-репо `digest --repos=a,b,c` — worker-pool

**Разрешение источника списка репозиториев** (совпадает с общим
приоритетом флаг > env > repo-config > personal-config, но с одним явным
уточнением): `--repos` **полностью заменяет** `digest.repos` из
`.gitl.yaml`, а не сливается с ним. Обоснование: `exclude_globs` в §5
специально **дополняет** (ADD), потому что это список исключений, где
объединение — безопасное поведение по умолчанию (больше исключений — не
навредит). Список репозиториев — противоположный случай: пользователь,
явно набравший `--repos=only-this-one`, ожидает digest **только** по
`only-this-one`, а не по нему плюс всё, что закоммичено в `.gitl.yaml`
командой, — слияние здесь было бы сюрпризом (случайно утёк бы отчёт по
чужим репозиториям в личный запуск). Это тот же принцип, что уже применён
к `--fail-on`/`--format` (флаг переопределяет **целиком**), а не к
`exclude_globs`; просто здесь явно проговаривается для списка.

- `digest.repos` в конфиге — новое поле `DigestConfig.Repos []RepoRef`
  (см. §10.6) в `internal/config`, наконец подключаемое в Этапе 3.
  Каждый элемент — `{path: "../service-a"}` (как в примере §5); путь
  резолвится **относительно repo-level `.gitl.yaml`**, если он найден
  (`filepath.Dir` найденного `.gitl.yaml`), иначе относительно текущей
  рабочей директории — то же место, откуда сам `.gitl.yaml` был прочитан,
  логично для относительных путей, объявленных внутри него.
- `--repos=a,b,c` — flat список путей через запятую (простые
  относительные/абсолютные ФС-пути от текущей рабочей директории запуска
  CLI, **не** от `.gitl.yaml` — флаг набирается руками в моменте, у него
  нет «дома», как у файла конфига).
- Если **ни флаг, ни конфиг** не задают список репозиториев — `digest`
  работает в **однорепо-режиме** над текущей директорией (эквивалент
  `--repos=.`) — не ошибка «--repos обязателен». Обоснование: `digest`
  без `--repos` — легитимный частый случай (см. §10.1), мульти-репо —
  надстройка, а не обязательное условие входа в команду.

**Worker-pool design:**

```go
// internal/gitlog/multidigest.go (package gitlog — обоснование в §10.6)

type RepoResult struct {
    Path  string
    Since time.Time
    Until time.Time
    Digest Digest // структура §10.2/§10.3 aggregation result
    Err   error   // непустой => остальные поля кроме Path нулевые
}

func CollectDigests(ctx context.Context, repoPaths []string, since time.Time, concurrency int) []RepoResult
```

- **Bounded worker pool** — фиксированное число воркеров `concurrency`
  (параметр, не бесконечный `go func()` на каждый репозиторий — репозиториев
  в реалистичном сценарии немного, но принцип «bounded», а не «unbounded
  fan-out», часть учебной цели горутин/каналов из PLAN.md §2). Дефолт
  `concurrency = runtime.GOMAXPROCS(0)`, зажатый сверху числом репозиториев
  (`min(GOMAXPROCS, len(repoPaths))`) — не заводим отдельный флаг
  `--concurrency` в Этапе 3 (не запрошено в скоупе, POST_MVP явно относит
  «расширенный горутинный worker-pool» к post-MVP — здесь минимально
  достаточная версия: фиксированный, не настраиваемый снаружи пул).
- Реализация — канал задач + `sync.WaitGroup`, не `errgroup` (сознательно:
  учебная цель — ручные каналы/`WaitGroup`, а не готовая библиотека,
  тот же принцип, что и «no LLM SDK» в §2 технологического стека):

  ```go
  func CollectDigests(ctx context.Context, repoPaths []string, since time.Time, concurrency int) []RepoResult {
      jobs := make(chan int)
      results := make([]RepoResult, len(repoPaths))

      var wg sync.WaitGroup
      for w := 0; w < concurrency; w++ {
          wg.Add(1)
          go func() {
              defer wg.Done()
              for i := range jobs {
                  results[i] = collectOne(ctx, repoPaths[i], since)
              }
          }()
      }
      for i := range repoPaths {
          jobs <- i
      }
      close(jobs)
      wg.Wait()
      return results
  }
  ```

  Запись в `results[i]` по индексу задачи, не через второй канал результатов
  — каждый индекс пишется ровно одним воркером ровно один раз, поэтому это
  безопасно без мьютекса (документируется комментарием в коде на этапе
  реализации) и даёт **устойчивый порядок вывода** (по исходному порядку
  `repoPaths`, не по порядку завершения) без отдельной сортировки —
  важно и для детерминизма, и для golden-тестов мульти-репо digest.
- **Context-отмена (Ctrl-C):** `ctx` протягивается в `collectOne` →
  `gitlog.NewRunner` → `exec.CommandContext`, как и везде в проекте (§3.1).
  Дополнительно, перед взятием задачи из канала `jobs`, воркер проверяет
  `ctx.Err()` — если контекст уже отменён, воркер не запускает `git`
  для оставшихся репозиториев и заполняет их `RepoResult.Err = ctx.Err()`,
  чтобы `Ctrl-C` не ждал, пока текущий батут задач дособерётся по одной.
  Более тонкий preemption (прерывание уже запущенного `git` процесса) не
  нужен отдельно — `exec.CommandContext` уже убивает процесс при отмене
  контекста, это стандартное поведение stdlib, не требующее кода здесь.
- **Изоляция ошибок на репозиторий:** любая ошибка на пути одного репо —
  директория не существует, не git-репозиторий, `git log` вернул ошибку,
  нет коммитов в окне у **этого конкретного** репо — не паникует и не
  прерывает остальные воркеры. Конкретно:
  - `gitlog.NewRunner(path)` не проверяет существование `path` сама по себе
    (`NewRunner` только проверяет `git` в `PATH`, не конкретную директорию) —
    ошибка «not a git repository» всплывёт из самого `git log` (git сам
    возвращает ненулевой код и понятный stderr на не-git директории), которую
    `Runner.run` уже оборачивает с текстом stderr (см. `runner.go`) —
    переиспользуется существующее поведение, не пишем отдельную проверку
    `os.Stat`.
  - `collectOne` перехватывает **любую** ошибку (`NewRunner`/`LogSince`)
    и кладёт её в `RepoResult.Err`, не паникует, не возвращает `error`
    вторым значением из `CollectDigests` — вызывающий код (`cli/digest.go`)
    решает, как показать ошибки (см. §10.5), а не прерывает остальные репо.
  - **Нет коммитов за окно у отдельного репо** — это **не** `RepoResult.Err`
    (это валидный результат, см. §10.1) — репо просто попадает в вывод
    с нулевой статистикой, как и в однорепо-режиме.
  - Никакого per-repo `context.WithTimeout` сверху общего `ctx` не вводится:
    единственный источник зависания — сам `git log`, у которого нет сетевого
    I/O (в отличие от LLM-вызовов, где таймаут уже есть в `llm.ClientConfig`),
    поэтому дополнительный таймаут был бы решением несуществующей проблемы
    в скоупе Этапа 3.

### 10.5 Агрегированный мульти-репо вывод

**Markdown:** каждый репозиторий — секция `## <path>`, с тем же телом,
что и однорепо-digest (§10.3), либо строкой ошибки; **плюс** финальная
секция `## Overall summary`, суммирующая только **успешные** репозитории:

```markdown
# Multi-repo digest — last 7 days (2026-06-29 → 2026-07-06)

## ../service-a

... (то же тело, что у однорепо digest) ...

## ../service-b

**Error:** not a git repository (or any of the parent directories)

## ../service-c

... ...

## Overall summary

- Repos requested: 3
- Repos OK: 2
- Repos failed: 1 (../service-b)
- Combined commits: 41
- Combined lines: +1204 / -390
```

- `Overall summary` **не** сливает `by_author`/`by_topic`/`top_files`
  across репозиториев в Этапе 3 (например, «Jane Doe» в двух репо не
  объединяется в одну строку с суммой) — сознательно оставлено как per-repo
  детализация; кросс-репо объединение авторов усложняется normalization
  проблемой (см. §10.2, не в скоупе) и не запрошено явным критерием
  приёмки Этапа 3 («агрегируется корректно» = не падает и показывает
  данные по каждому репо + верхнеуровневые числа, не поимённое слияние).
- **JSON** — структурно расширяет §10.3 массивом `repos` + верхнеуровневым
  `overall`:

```json
{
  "schema_version": 1,
  "command": "digest",
  "generated_at": "2026-07-06T09:00:00Z",
  "days": 7,
  "since": "2026-06-29T09:00:00Z",
  "until": "2026-07-06T09:00:00Z",
  "repos": [
    {
      "path": "../service-a",
      "ok": true,
      "error": "",
      "stats": {"commits": 23, "files_changed": 14, "lines_added": 812, "lines_removed": 145},
      "by_author": [{"author": "Jane Doe", "commits": 14, "lines_added": 610, "lines_removed": 90}],
      "by_topic": [{"topic": "feat", "commits": 10}],
      "top_files": [{"path": "internal/llm/client.go", "commits": 5}]
    },
    {
      "path": "../service-b",
      "ok": false,
      "error": "not a git repository (or any of the parent directories)",
      "stats": null,
      "by_author": null,
      "by_topic": null,
      "top_files": null
    }
  ],
  "overall": {
    "repos_requested": 3,
    "repos_ok": 2,
    "repos_failed": 1,
    "combined_commits": 41,
    "combined_lines_added": 1204,
    "combined_lines_removed": 390
  }
}
```

- Однорепо и мульти-репо JSON — **одна и та же схема**, не два разных
  `schema_version`-пространства: однорепо-вызов — это `repos` с одним
  элементом и `overall` с `repos_requested/ok = 1`. Обоснование: избегать
  двух параллельных JSON-контрактов для «по сути одной и той же команды
  с разным числом входов» — консьюмеру (например, CI-скрипту) не нужно
  разбирать, какой из двух форматов пришёл, если он всегда обрабатывает
  `repos[]`. Однорепный `--format=json` без `--repos`/`digest.repos` печатает
  тот же конверт с `repos: [{"path": ".", ...}]`.
- `ok: false` → все агрегационные поля этого элемента — `null` (не пустые
  массивы/нулевые числа) — явно отличимо от «репозиторий без коммитов за
  окно» (`ok: true`, `stats.commits: 0`, пустые массивы `[]`), что важно
  для консьюмера, который должен по-разному обработать «нет данных» и
  «сбой сбора».

### 10.6 Точки изменения в коде (ориентир)

- `internal/config/config.go`: новый `DigestConfig` в `Config`:

  ```go
  type Config struct {
      LLM    LLMConfig
      Cost   CostConfig
      Output OutputConfig
      Diff   DiffConfig
      Policy PolicyConfig
      Digest DigestConfig `mapstructure:"digest"`
  }

  type DigestConfig struct {

  ```

  Добавляется в `defaults()` как отсутствующий ключ (нет built-in дефолтных
  репозиториев — пустой список означает «однорепо-режим», см. §10.4).
- `internal/gitlog/runner.go`: `Source` расширяется методами
  `LatestTag(ctx) (string, error)`, `LogSince(ctx, since time.Time) ([]Commit, error)`,
  `DiffForCommit(ctx, hash string) (string, error)` — все три на существующем
  `Runner`, тем же паттерном `r.run(ctx, "git", args...)`, что и `Log`/`Diff`.
- Новый `internal/gitlog/changelog.go` — категоризация (§9.2, §9.5).
- Новый `internal/gitlog/digest.go` — однорепо-агрегация (§10.2):
  `AggregateDigest(commits []Commit, diffs map[string]string, since, until time.Time) Digest`
  (чистая функция; `diffs` — map hash→diff для построчной статистики по
  автору, собираемая вызывающим кодом через `DiffForCommit` до вызова).
- Новый `internal/gitlog/multidigest.go` — worker-pool (§10.4):
  `CollectDigests(ctx, repoPaths []string, since time.Time, concurrency int) []RepoResult`.
  Решение поместить worker-pool в `gitlog`, а не в `cli/digest.go`:
  worker-pool оперирует **на уровне репозиториев git-данных** (создаёт
  `Runner` на каждый путь, гоняет `Log`/`Diff`), то есть по своей природе —
  это тот же слой ответственности, что и однорепо-сбор, просто
  распараллеленный; `cli/digest.go` остаётся тонким (парсинг флагов →
  вызов `gitlog`/`render`), как и `review.go`/`changelog.go` — граница
  пакетов «cli = command wiring, gitlog = git data» (из ТЗ задачи) держится
  ровно так же, как и для остальных команд.
- Новый `internal/render/digest.go` (пакет `render`, не новый пакет —
  то же обоснование, что в §9.6): тип `DigestArtifact`, константа
  `DigestSchemaVersion = 1`, функция
  `RenderDigest(w io.Writer, art DigestArtifact, format Format) error`.
  `DigestArtifact` уже структурно несёт `Repos []RepoDigest` (даже
  в однорепо-случае — один элемент), чтобы не заводить два Go-типа под
  одну JSON-схему (§10.5).
- Новые команды `internal/cli/changelog.go`, `internal/cli/digest.go` —
  по файлу на команду (существующая конвенция), регистрируются в
  `newRootCmd` рядом с `newReviewCmd`.

## 11. Golden-тесты рендера (детали Этапа 3)

> Стратегия golden-тестирования для трёх рендер-путей (`review` уже покрыт
> field-by-field юнит-тестами, см. `internal/render/render_test.go`
> `TestRenderJSONSchema` — этот подход **не переиспользуется как есть** для
> `changelog`/`digest`, см. обоснование ниже).

### 11.1 Почему golden-файлы, а не field-by-field, для changelog/digest

`review`'s JSON-тест сознательно избегает golden-файлов, потому что
`Artifact.GeneratedAt` в реальном прогоне — `time.Now()`, и один и тот же
вход не даёт побайтово одинаковый вывод между прогонами. Это же ограничение
в равной мере касается `changelog`/`digest`: их JSON и Markdown тоже несут
`generated_at`. Решение — **не архитектурное отличие двух подходов**,
а единая техника инъекции фиксированного времени (см. §11.3), после которой
golden-сравнение становится побайтово стабильным. Раз проблема решена на
уровне тестового харнесса, а не архитектуры рендера, golden-файлы становятся
предпочтительным способом для changelog/digest — они переносят «эталонный
вывод» из кода теста (десятки `if got != want`) в читаемый артефакт
в `testdata/`, что для многострочных Markdown-таблиц и многополевого JSON
значительно снижает трение при добавлении новых полей (диффом в PR виден
весь эффект изменения на реальном примере, не только те поля, которые
кто-то не забыл проверить в assertion). `review` не переписывается на
golden в Этапе 3 (не входит в скоуп задачи — «не трогать другие команды»),
но новый паттерн для changelog/digest фиксируется здесь как канонический
для всех **будущих** рендеров.

### 11.2 `testdata/` layout и механизм `-update`

```
internal/render/
├── render.go
├── render_test.go          # существующий, без изменений в Этапе 3
├── changelog.go
├── changelog_test.go        # новый, golden-based
├── digest.go
├── digest_test.go           # новый, golden-based
└── testdata/
    ├── changelog/
    │   ├── basic.md
    │   ├── basic.json
    │   ├── basic.txt
    │   ├── breaking.md      # покрывает секцию ⚠ BREAKING CHANGES
    │   ├── breaking.json
    │   ├── missing_required.md   # покрывает предупреждение §9.3 в самом выводе, если применимо, либо только stderr — уточняется в тесте
    │   └── empty.md          # 0 коммитов в диапазоне
    ├── digest/
    │   ├── single_repo.md
    │   ├── single_repo.json
    │   ├── single_repo.txt
    │   ├── multi_repo.md
    │   ├── multi_repo.json
    │   └── multi_repo_with_error.md   # один из репо ok:false
```

- Golden-файлы лежат **рядом с пакетом**, который их проверяет
  (`internal/render/testdata/...`), а не в корневом `testdata/` из
  `TECHNICAL_PLAN.md §3` — уточнение верхнеуровневой схемы: корневой
  `testdata/` из §3 остаётся для **git-фикстур** (заранее собранные
  репозитории/сырой вывод `git log`, если такие когда-то понадобятся для
  парсера); golden-файлы **рендер-вывода** идиоматично лежат в
  `<пакет>/testdata/`, что и так стандартная Go-конвенция
  (`go test` игнорирует `testdata/` при сборке автоматически по имени
  каталога, независимо от глубины).
- **`-update`-флаг** — ровно идиоматичный Go-паттерн, пакетный var:

  ```go
  var update = flag.Bool("update", false, "update golden files in testdata/")
  ```

  в `changelog_test.go` (или в новом `internal/render/golden_test.go`,
  если флаг захочется расшарить между `changelog_test.go`/`digest_test.go`
  без дублирования — второй вариант предпочтителен, т.к. у нас сразу два
  потребителя одного флага, что уже оправдывает общий helper-файл, не
  нарушая «интерфейс на второй реализации»: здесь не интерфейс, а обычная
  переиспользуемая тестовая утилита).
  Общий helper в `internal/render/golden_test.go`:

  ```go
  var update = flag.Bool("update", false, "update golden files in testdata/")

  func assertGolden(t *testing.T, path string, got []byte) {
      t.Helper()
      if *update {
          if err := os.MkdirAll(filepath.Dir(path), 0o755); err != nil {
              t.Fatal(err)
          }
          if err := os.WriteFile(path, got, 0o644); err != nil {
              t.Fatal(err)
          }
          return
      }
      want, err := os.ReadFile(path)
      if err != nil {
          t.Fatalf("read golden %s: %v (run with -update to create it)", path, err)
      }
      if !bytes.Equal(got, want) {
          t.Errorf("output does not match golden %s\n--- got ---\n%s\n--- want ---\n%s", path, got, want)
      }
  }
  ```

  Обновление: `go test ./internal/render/... -run TestChangelogGolden -update`
  (или без `-run`, чтобы обновить всё разом — оба варианта работают,
  документируется в комментарии над `var update`).
- Сравнение — **побайтовое** (`bytes.Equal`), как и предписано в SKILL.md
  §4 («сравнение — байт-в-байт»); никакой нормализации whitespace на
  стороне теста.

### 11.3 Обработка недетерминированных полей (`generated_at`)

**Единый подход для changelog и digest: инъекция фиксированного времени
через явный параметр артефакта, не через глобальные часы/интерфейс `Clock`.**
Оба новых `ChangelogArtifact`/`DigestArtifact` (§9.6, §10.6) уже получают
`GeneratedAt time.Time` как обычное поле структуры — ровно как
`render.Artifact.GeneratedAt` для review — и тесты рендер-пакета
конструируют артефакт напрямую (`ChangelogArtifact{GeneratedAt: time.Date(2026, 7, 6, 9, 0, 0, 0, time.UTC), ...}`),
минуя реальный `time.Now()` целиком. Это уже устраняет недетерминизм
на уровне golden-теста `internal/render`: рендер-функции здесь тестируются
в отрыве от `cli`-слоя, где реальный `time.Now()` вызывается один раз
(в `cli/changelog.go`/`cli/digest.go`, аналогично `runReview`) — golden-тесты
рендера этот код не проходят вовсе, так что вопрос «мокать часы» на этом
уровне не возникает в принципе (это то же самое, почему `render_test.go`'s
`TestRenderJSONSchema` уже может себе позволить фиксированный `GeneratedAt`
в `sampleArtifact()` без всякого мокания).

Отдельный вопрос — **интеграционные тесты `cli/changelog_test.go`/
`cli/digest_test.go`**, которые (как и существующий `cli/review_test.go`)
гоняют команду целиком поверх временного git-репозитория и **не** могут
заранее знать `generated_at`, потому что его выставляет `time.Now()` внутри
`runChangelog`/`runDigest`. Для этого уровня — **не golden-файлы**, тот же
подход, что уже применён в `review_test.go`: `strings.Contains`-проверки
на присутствие ожидаемых секций/полей (`"## Added"`, `"schema_version": 1`,
конкретные хэши/темы, которые детерминированы от фикстур репозитория) —
`generated_at` в этих тестах просто не проверяется побайтово. Явное правило,
которым руководствуется реализация: **golden byte-for-byte сравнение — только
в `internal/render/*_test.go`, где вход собирается вручную и `GeneratedAt`
фиксирован**; **`internal/cli/*_test.go` — content-assertion стиль**, как
и сегодня для review, потому что там неизбежен реальный `time.Now()`. Это
не два принципа, а один: «побайтовое сравнение допустимо только там, где
100% входа детерминировано инъекцией» — тот же критерий, применённый
дважды на двух разных уровнях пирамиды тестов.

### 11.4 Что именно покрывается golden-тестами

- `internal/render/changelog_test.go`: `TestChangelogGoldenMarkdown`,
  `..._JSON`, `..._Text` над `basic`-фикстурой (несколько категорий,
  включая пустые); отдельный `TestChangelogGoldenBreaking` для
  `⚠ BREAKING CHANGES`-секции; `TestChangelogGoldenEmpty` для 0 коммитов
  в диапазоне (проверяет, что рендер не паникует и печатает осмысленный
  «no commits» текст вместо пустых заголовков секций).
- `internal/render/digest_test.go`: `TestDigestGoldenSingleRepo` (md/json/text);
  `TestDigestGoldenMultiRepo` (несколько репо, все успешны);
  `TestDigestGoldenMultiRepoWithError` (один `ok:false` репозиторий —
  проверяет и Markdown-секцию ошибки, и `null`-поля в JSON, см. §10.5).
- `-race` обязателен для нового пакетного теста
  `internal/gitlog/multidigest_test.go` (`go test -race ./internal/gitlog/...`)
  — единственное место конкурентности в Этапе 3 (по прямому требованию
  задачи — «не добавлять worker-pool в review/changelog»); тест гоняет
  `CollectDigests` над 3 временными репозиториями (один — не git-репозиторий,
  для проверки graceful degradation, см. критерий приёмки Этапа 3) и
  над отменённым контекстом (Ctrl-C-сценарий) — оба сценария содержательно
  требуются критерием приёмки TECHNICAL_PLAN §6 «не падает, если один из
  репо недоступен».

## 12. Риски и как их обойти (§6)

| Риск | Проявление | Обход |
|---|---|---|
| **Лимит токенов LLM на объёмном диффе** | Большой релиз не влезает в контекст → обрыв/ошибка/мусор | Порог `max_diff_bytes`; `exclude_globs`; стратегия «сначала `--stat` + сообщения коммитов, полный дифф — только для топ-N крупных файлов»; при переполнении — чанкинг по файлам с map-reduce агрегацией; явный маркер `[... diff truncated ...]`. |
| **Качество AI-сводки на шумных диффах** | Автоформатер/локи забивают ревью | `exclude_globs` по умолчанию (lock/min.js/vendor/svg); низкая `temperature` (0.2); промпт с инструкцией игнорировать форматирование; validation-first ловит это на Этапе 1. |
| **Недетерминизм → хрупкие тесты** | LLM-вывод нельзя проверить golden-файлом | Тестировать **всё, кроме** самого вызова LLM: парсер, рендер, retry (через `httptest`), сборку промпта. LLM-слой — за интерфейсом, подменяется offline/мок-провайдером. |
| **Кросс-платформенные грабли** | Пути/конфиг ломаются на Windows | `os.UserConfigDir()`/`filepath.Join` вместо хардкода; goreleaser собирает под все ОС; ранний smoke-тест бинаря под Windows (`--help`). |
| **Отсутствие `git` в PATH** | Паника/непонятная ошибка | Проверка `exec.LookPath("git")` при старте, дружелюбное сообщение с инструкцией. |
| **Утечка приватного диффа в LLM** | Пользователь шлёт закрытый код в чужой API | Документация в README; локальные/enterprise провайдеры (Ollama/Azure OpenAI) для резидентности; никакой телеметрии. |
| **Ложное доверие к AI-ревью при жёстком гейтинге** | `--fail-on=high` даёт false positive → блокирует мерж → инструмент выключают | Дефолт — **WARN, не BLOCK** (жёсткий gate — явный opt-in); консервативная калибровка риск-скоринга (Этап 2); ревью всегда показывает reasoning, а не только вердикт. |
| **Секреты в CI** | API-ключ утекает в логи/артефакты Action | Документация `secrets.*`-паттерна; ключ никогда не логируется даже в verbose; тест на отсутствие ключа в stdout/stderr при ошибках. |
| **Cost-guard не срабатывает** | Баг в оценке стоимости → незаметный крупный счёт | `--dry-run`/`--max-cost-usd` — отдельно тестируемая логика (unit-тесты на границы); консервативный дефолт; явное предупреждение при отключении guard. |
| **Неподписанный артефакт не пускают в CI** | Строгие CI-политики отклоняют неподписанные бинари | Cosign-подпись + `SHA256SUMS` — в MVP (Этап 5); задокументированная процедура верификации. |
| **Выгорание интереса** | Проект брошен на середине | Не решается урезанием скоупа — решается validation-first (Этап 1 даёт рабочее ядро рано) и чёткими per-этап deliverable с критерием «зелёно/не зелёно»; личная польза (dogfooding) — встроенная мотивация. |
| **Over-engineering ради «красоты»** | Абстракции, которых никто не просил | Правило «интерфейс появляется на второй реализации, не на первой»; `pkg/` не заводить, пока не попросят внешние потребители. |
