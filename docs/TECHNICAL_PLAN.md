## 1. Вводные и ограничения

- Соло-разработчик, реализация с AI-ассистентом; приоритет — обучение Go
  (нагружать язык и stdlib, а не SDK).
- Целевые инфраструктурные расходы ≈ **$0/мес**; экономическая обоснованность
  необязательна.
- Требуется установленный системный `git` в `PATH` (для `os/exec`) — приемлемо,
  т.к. аудитория — разработчики.
- LLM обрабатывает **текст диффа** → жёсткий потолок по токенам → нужна стратегия
  усечения/чанкинга (см. §6, «Лимит токенов»).
- **Go 1.22+** (нужны `slices`, `maps`, `log/slog`, современный `errors.Join`).
- Кросс-платформенность: пути и конфиг через `os.UserConfigDir()`, а не хардкод.
- Без телеметрии и «звонков домой» — принципиально (доверие + приватность диффов).

## 2. Технический стек (§4.1 первоисточника)

| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | Go 1.22+ | Приоритет обучения; stdlib закрывает ~90% нужд |
| CLI | `spf13/cobra` | Стандарт индустрии (kubectl, gh, hugo), чистое дерево команд/флагов |
| Конфиг | `spf13/viper` | YAML + env + флаги в единой иерархии, интеграция с cobra |
| Git | системный `git` через `os/exec` | см. §4 |
| HTTP к LLM | `net/http` (stdlib) | Осознанно **без** SDK — тренировка ручного клиента, стриминга, retry |
| JSON | `encoding/json` (stdlib) | Парсинг ответов LLM и `--format=json` |
| TUI (opt, post-MVP) | `charmbracelet/bubbletea` + `lipgloss` | Только для `digest --tui`, поверх готовых артефактов |
| Логи | `log/slog` (stdlib) | Структурные логи, `--verbose` → debug |
| Тесты | stdlib `testing` + table-driven | Идиоматика Go; golden-файлы для рендеров |
| Сборка/релиз | `goreleaser` | Кросс-компиляция + GitHub Releases + Homebrew одним конфигом |

Сознательное ограничение зависимостей: git — stdlib `os/exec`, HTTP — stdlib
`net/http`. Внешние зависимости только там, где они дают обучающий каркас
(cobra/viper/bubbletea) или несоразмерны для ручной реализации (goreleaser).

**Почему `internal/`:** всё ядро в `internal/`, чтобы никто не импортировал
незрелый API как библиотеку — идиоматичный способ пометить «это приложение,
не библиотека». **`pkg/` намеренно не заводим на старте**: публичный API появится,
только если будет спрос («не строй библиотеку, пока её не просят»).

### 3.1 Поток данных (одна команда, §4.5)

```
cobra cmd → config.Load() → gitlog.Runner.Log/Diff(ctx, range)
   → prompt.Build(commits, diff, template)
   → llm.Client.Complete(ctx, prompt)  [retry/backoff, offline-fallback]
   → render.Render(artifact, format)   → stdout/file/TUI
```

Отмена по Ctrl-C — единый `context.Context` из `signal.NotifyContext`,
протянутый через exec и HTTP.

## 4. `os/exec` vs `go-git` — решение (§4.3)

**Выбор: системный `git` через `os/exec`.** Причины, релевантные цели
«написание + обучение Go»:

1. **Тренирует ядровой Go-навык** — работу с внешними процессами:
   `exec.CommandContext`, чтение `stdout`/`stderr` через `bytes.Buffer`/
   `bufio.Scanner`, коды возврата, контекст-отмена.
2. **Заставляет писать парсер** — самая обучающая часть: спроектировать
   `--pretty=format:` с разделителями, распарсить `--name-status`, покрыть
   table-driven-тестами на golden-выводе. С `go-git` данные приходят структурами —
   упражнение исчезает.
3. **Простая ментальная модель** — поведение 1:1 с терминалом; отлаживается
   тривиально (скопировал команду, запустил вручную).
4. **`go-git` тяжелее и с нюансами** — не 100% паритет с CLI по краевым случаям,
   больше зависимость.

Компромисс инкапсулируется: весь `os/exec` спрятан за интерфейсом в
`internal/gitlog` — если позже захочется `go-git`, меняется одна реализация,
команды не трогаются.

**Парсинг git log с control-разделителями (критичная деталь):**

```
git log --pretty=format:%H%x1f%an%x1f%aI%x1f%s%x1f%b%x1e --name-status <range>
```

- `%x1f` (ASCII **unit separator**, 0x1F) разделяет **поля** внутри коммита.
- `%x1e` (ASCII **record separator**, 0x1E) разделяет **коммиты**.

Control-символы из ASCII-зоны 0x1F/0x1E не встречаются в сообщениях коммитов,
поэтому надёжны как разделители. **Частая ошибка новичка — split по `\n`**: тело
коммита содержит и переводы строк, и пробелы, поэтому наивный split ломается.

Целевые типы (`internal/gitlog/types.go`):

```go
type Commit struct {
    Hash    string
    Author  string
    Date    time.Time
    Subject string
    Body    string
    Files   []FileChange
}
type FileChange struct {
    Status string // A, M, D, R...
    Path   string
    Old    string // для переименований (R)
}
```

## 5. Схема конфигурации (§4.4)

Два уровня, сливаются по приоритету
**флаг > env > `.gitl.yaml` (репозиторий) > `~/.config/gitl/config.yaml` (личный)**.
Личный файл — путь через `os.UserConfigDir()` (на Windows → `%AppData%\gitl`);
repo-level `.gitl.yaml` в корне репозитория коммитится в git и является общей
политикой команды.

```yaml
# ~/.config/gitl/config.yaml — личные настройки (секреты, провайдер по умолчанию)
llm:
  provider: "openai"          # "openai" | "ollama" | "azure_openai" — явный контракт
  api_key: ""                 # или env GITL_API_KEY (рекомендовано, особенно в CI)
  base_url: "https://api.openai.com/v1"
  model: "gpt-4o-mini"
  max_tokens: 1500
  temperature: 0.2
  timeout_seconds: 60
  max_retries: 3
  # azure_openai: endpoint, deployment, api_version — под-блок при provider=azure_openai
  # ollama: host (по умолчанию http://localhost:11434) — без api_key

cost:
  max_cost_usd: 0.50          # cost-guard: отказ выполнять запрос дороже порога
  warn_at_usd: 0.10           # предупреждение, но не блокировка

output:
  format: "md"                # md | text | json
  color: true

diff:
  max_diff_bytes: 120000
  exclude_globs:
    - "*.lock"
    - "*.min.js"
    - "vendor/**"
    - "*.svg"
```

```yaml
# .gitl.yaml — repo-level политика (коммитится в репозиторий, общая для команды)
policy:
  fail_on: "high"             # low | medium | high | never — порог гейтинга CI
  required_changelog_categories: ["Added", "Fixed", "Security"]
  exclude_globs: ["migrations/**"]   # ДОБАВЛЯЕТСЯ к личным exclude_globs, не заменяет

digest:
  repos:                      # мульти-репо digest
    - path: "../service-a"
    - path: "../service-b"
```

**Валидация при старте:** если `api_key` пуст/недоступен → offline-режим
с предупреждением в stderr (не падаем). Провайдер выбирается явным полем
`provider`, а не угадыванием по `base_url` — это и есть контракт
мультипровайдерности.

## 7. Контракт риск-скоринга и версионированного JSON-вывода (детали Этапа 2)

> Дизайн-решения, зафиксированные **до** реализации Этапа 2, чтобы не изобретать
> схему по ходу кодирования. Дополняет §6 «Этап 2».

### 7.1 Как модель сообщает риск

Однократный вызов LLM, без второго round-trip ради скоринга (дороже и сложнее
retry-логики). `reviewSystem`-промпт (`internal/prompt/templates.go`) получает
четвёртый обязательный пункт вдобавок к Summary/Notable changes/Concerns:

> 4. Закончи ответ отдельным fenced-блоком с языком `risk`, содержащим ровно
> один однострочный JSON-объект, и это должно быть последним, что ты выводишь:
> `{"level": "low|medium|high", "summary": "<одно предложение, до 140 символов>"}`.

Пример хвоста ответа модели:

````
## Concerns

- ...

```risk
{"level": "medium", "summary": "Touches auth middleware without new tests."}
```
````

### 7.2 Парсинг и fallback

- Ищем **последний** fenced-блок с языком `risk` (терпимо принимаем и ```json,
  если внутри валидный объект с ключом `level`); `level` приводим к нижнему
  регистру и валидируем как `low|medium|high`.
- Валидный блок вырезается из текста перед рендером — пользователь не должен
  видеть сырой JSON в Markdown/text-выводе.
- Если блока нет / JSON невалиден / `level` не из трёх значений — эвристический
  fallback (§7.3), с `slog.Warn` о том, что оценка модели не распарсилась.
- **Offline-провайдер всегда** использует эвристику (нет модели, которая могла
  бы дать риск) — он и раньше это делал по духу, теперь формально возвращает
  структурированное поле, а не только текст.

### 7.3 Эвристика (offline-режим и fallback при сбое парсинга)

Одна детерминированная функция на оба случая (одинаковый вход → одинаковый
выход, без сети и случайности — правило офлайн-провайдера из SKILL.md):

- **high**, если diff задевает чувствительные пути/ключевые слова
  (`auth`, `secret`, `password`, `token`, `credential`, `security`, `payment`,
  `migration`, `permission` — регистронезависимый поиск по путям файлов,
  темам и телам коммитов) **или** изменённых строк больше ~300, **или**
  затронуто больше ~20 файлов.
- **low**, если изменённых строк меньше ~20, файлов ≤ 3 и чувствительных
  совпадений нет.
- **medium** — всё остальное (дефолт).
- `summary` — короткое шаблонное объяснение сработавшего правила (например,
  `"Large diff (312 changed lines) across 14 files."` или
  `"Touches security-sensitive keywords: \"auth\", \"token\"."`).

Точные пороги — не догма, их можно откалибровать при реализации; важна
детерминированность и то, что offline и online-fallback используют одну
и ту же функцию (не две параллельные реализации).

### 7.4 JSON-схема вывода (`--format=json`, `schema_version: 1`)

```json
{
  "schema_version": 1,
  "generated_at": "2026-07-05T18:32:10Z",
  "range": "HEAD~5..HEAD",
  "offline": false,
  "provider": "openai",
  "model": "gpt-4o-mini",
  "risk": {
    "level": "medium",
    "summary": "Touches auth middleware without new tests."
  },
  "stats": {
    "commits": 5,
    "files_changed": 12,
    "lines_added": 340,
    "lines_removed": 58
  },
  "commits": [
    {
      "hash": "abcd123",
      "author": "Jane Doe",
      "date": "2026-07-01T10:00:00Z",
      "subject": "feat: add token refresh"
    }
  ],
  "review_markdown": "## Summary\n...\n## Notable changes\n...\n## Concerns\n..."
}
```

Правила версии: `schema_version` — целое число, бампается только при breaking
change (переименование/удаление поля, смена типа); добавление нового
необязательного поля версию не меняет. `review_markdown` — тот же текст, что
идёт в `--format=md` (без хвостового `risk`-блока, он уже вынесен в `risk`);
`--format=text` — тот же контент с грубо срезанной Markdown-разметкой
(`#`/`**`/бэктики), без отдельного JSON-парсинга. `risk` всегда заполнено
(offline и fallback гарантируют значение) — консьюмеру в CI не нужно
обрабатывать `null`.

### 7.5 Точки изменения в уже написанном коде Этапа 1 (ориентир)

- `llm.Request` получает `Commits []gitlog.Commit` и `Diff string` —
  pass-through, чтобы эвристика (§7.3) считалась внутри провайдера без
  повторного парсинга текста промпта; `llm.Response` получает поле `Risk`.
- Общая `llm.HeuristicRisk(commits, diff) Risk` — вызывается и из `Offline`,
  и как fallback из `Client` при нераспарсенном риск-блоке.
- `render.Render` меняет вход с голой строки на артефакт, несущий поля §7.4
  (range/risk/stats/commits/offline/provider/model/review_markdown); md-рендер
  добавляет короткую шапку `**Risk:** MEDIUM — summary` перед текстом ревью.
  Конкретные имена Go-типов — на усмотрение реализации, wire-контракт JSON
  выше — обязателен.
- Таблица цен для `--dry-run`/`--max-cost-usd` в этот дизайн не входит —
  отдельное решение перед стартом Этапа 2.

## 8. Оценка стоимости и cost-guard (детали Этапа 2)

> Дизайн `--dry-run`/`--max-cost-usd`, зафиксированный до реализации. Дополняет
> §6 «Этап 2». Без реального токенайзера и без вызова стороннего pricing-API —
> в духе «$0/мес инфраструктуры» и офлайн-совместимости.

### 8.1 Оценка токенов

Грубая эвристика, не точный расчёт (не `tiktoken`): `EstimateTokens(text) =
ceil(len(text_in_bytes) / 4)` — общепринятое приближение «~4 символа на
токен». Считается по итоговому промпту (system+user), который реально уйдёт
в API. Выходные токены не гадаем по содержанию — берём сконфигурированный
`llm.max_tokens` как потолок (это и есть то число, что уходит в запрос).
Итоговая оценка токенов умножается на **×1.15** — сознательный запас в сторону
переоценки: ошибка "посчитали дороже, чем есть" безопасна (просто просит
поднять лимит), ошибка "посчитали дешевле" — именно то, от чего guard должен
защищать.

### 8.2 Таблица цен + override

Встроенная таблица (например, `internal/llm/cost.go`), цены **за 1M токенов**,
ориентировочные на момент написания плана (2026-07, сверено с
developers.openai.com/api/docs/pricing и сторонними агрегаторами) —
устаревание ожидаемо и не является багом, но стоит перепроверить перед
реализацией:

| Модель | Input / 1M | Output / 1M |
|---|---|---|
| `gpt-4o-mini` (дефолт `llm.model`) | $0.15 | $0.60 |
| `gpt-4o` | $2.50 | $10.00 |
| `gpt-4.1-mini` | $0.40 | $1.60 |

- **Explicit override** в конфиге: `cost.price_per_1m_input` /
  `cost.price_per_1m_output` (float64, `0` = использовать встроенную таблицу).
  Перекрывает таблицу безусловно — нужен для моделей не из таблицы,
  устаревших цен без правки кода, self-hosted OpenAI-совместимых эндпоинтов
  с другим прайсингом.
- **`provider: ollama`** — стоимость всегда `$0` (self-hosted); guard/оценка
  пропускаются целиком, без предупреждений (ожидаемый бесплатный путь).
- **Модель не найдена в таблице и override не задан** (провайдер не ollama) —
  guard **не блокирует**: явное предупреждение в stderr
  (`no pricing data for model %q — cost estimate unavailable, proceeding
  without cost-guard (set cost.price_per_1m_input/output to enable it)`) и
  выполнение продолжается без проверки. Permissive-by-default — кастомная
  модель не должна молча падать из-за отсутствия в таблице.

### 8.3 `--dry-run`

- Никакого сетевого вызова. Печатает в **stdout** (это полезная нагрузка
  команды, не диагностика): оценку input-токенов, потолок output-токенов,
  итоговую оценку в USD, модель/провайдера и явную пометку «estimate, not
  exact» — и завершает работу с exit code 0, не вызывая провайдера.
- В offline-режиме (`cfg.OfflineMode()`) `--dry-run` печатает «offline mode —
  no API call, no cost» и выходит 0, а не падает с ошибкой.

### 8.4 `--max-cost-usd` guard

- Считается тем же путём, что и `--dry-run`, но встроен в обычный `review`
  (не отдельная команда), выполняется **до** вызова провайдера, сразу после
  сборки промпта.
- `cost.max_cost_usd` из конфига, переопределяется флагом `--max-cost-usd`
  (через уже существующий механизм `bindChangedFlags`).
- `estimate > max_cost_usd` → ошибка, ненулевой exit code, сообщение с
  цифрами и путями решения, например:
  `gitl: estimated cost $0.42 (≈14,231 input + 1,500 output tokens @
  gpt-4o-mini) exceeds --max-cost-usd=0.01 — increase the limit, use
  --dry-run to inspect, or omit the API key for a free offline review.`
- `warn_at_usd < estimate ≤ max_cost_usd` → предупреждение в stderr,
  выполнение продолжается.
- `max_cost_usd <= 0` — guard явно отключён («без лимита»); это осознанный
  выбор пользователя (документируется в README), не дефолт (дефолт остаётся
  `$0.50` из конфига Этапа 1).
- Guard полностью пропускается в offline-режиме — там нет вызова и нет
  стоимости.

### 8.5 Точки изменения в коде (ориентир)

- Новый `internal/llm/cost.go`: `EstimateTokens`, встроенная таблица цен,
  `EstimateCost(...) (Estimate, error)` — чистые, детерминированные функции;
  юнит-тесты на границы, включая тест, прямо воспроизводящий Критерий успеха
  Этапа 2: `--max-cost-usd=0.01` блокирует крупный синтетический дифф.
- `config.CostConfig` получает `PricePer1MInput`/`PricePer1MOutput`
  (`mapstructure: price_per_1m_input`/`price_per_1m_output`).
- `cli/review.go`: флаги `--dry-run` (bool) и `--max-cost-usd` (float64,
  добавлен в `bindChangedFlags` → `cost.max_cost_usd`); guard вызывается
  после сборки промпта (нужен реальный текст запроса), до `selectProvider`/
  `provider.Complete`.

## 9. Риски и как их обойти (§6)

| Риск | Проявление | Обход |
|---|---|---|
| **Лимит токенов LLM на объёмном диффе** | Большой релиз не влезает в контекст → обрыв/ошибка/мусор | Порог `max_diff_bytes`; `exclude_globs`; стратегия «сначала `--stat` + сообщения коммитов, полный дифф — только для топ-N крупных файлов»; при переполнении — чанкинг по файлам с map-reduce агрегацией; явный маркер `[... diff truncated ...]`. |
| **Качество AI-сводки на шумных диффах** | Автоформатер/локи забивают ревью | `exclude_globs` по умолчанию (lock/min.js/vendor/svg); низкая `temperature` (0.2); промпт с инструкцией игнорировать форматирование; validation-first ловит это на Этапе 1. |
| **Недетерминизм → хрупкие тесты** | LLM-вывод нельзя проверить golden-файлом | Тестировать **всё, кроме** самого вызова LLM: парсер, рендер, retry (через `httptest`), сборку промпта. LLM-слой — за интерфейсом, подменяется offline/мок-провайдером. |
| **Кросс-платформенные грабли** | Пути/конфиг ломаются на Windows | `os.UserConfigDir()`/`filepath.Join` вместо хардкода; goreleaser собирает под все ОС; ранний smoke-тест бинаря под Windows (`--help`). |
| **Отсутствие `git` в PATH** | Паника/непонятная ошибка | Проверка `exec.LookPath("git")` при старте, дружелюбное сообщение с инструкцией. |
| **Утечка приватного диффа в LLM** | Пользователь шлёт закрытый код в чужой API | Документация в README; локальные/enterprise провайдеры (Ollama/Azure OpenAI) для резидентности; никакой телеметрии. |
| **Ложное доверие к AI-ревью при жёстком гейтинге** | `--fail-on=high` даёт false positive → блокирует мерж → инструмент выключают | Дефолт — **WARN, не BLOCK** (жёсткий gate — явный opt-in); консервативная калибровка риск-скоринга (Этап 2); ревью всегда показывает reasoning, а не только вердикт. |
| **Секреты в CI** | API-ключ утекает в логи/артефакты Action | Документация `secrets.*`-паттерна; ключ никогда не логируется даже в verbose; тест на отсутствие ключа в stdout/stderr при ошибках. |
| **Cost-guard не срабатывает** | Баг в оценке стоимости → незаметный крупный счёт | `--dry-run`/`--max-cost-usd` — отдельно тестируемая логика (unit-тесты на границы); консервативный дефолт; явное предупреждение при отключении guard. |
| **Неподписанный артефакт не пускают в CI** | Строгие CI-политики отклоняют неподписанные бинари | Cosign-подпись + `SHA256SUMS` — в MVP (Этап 5); задокументированная процедура верификации. |
| **Выгорание интереса** | Проект брошен на середине | Не решается урезанием скоупа — решается validation-first (Этап 1 даёт рабочее ядро рано) и чёткими per-этап deliverable с критерием «зелёно/не зелёно»; личная польза (dogfooding) — встроенная мотивация. |
| **Over-engineering ради «красоты»** | Абстракции, которых никто не просил | Правило «интерфейс появляется на второй реализации, не на первой»; `pkg/` не заводить, пока не попросят внешние потребители. |
