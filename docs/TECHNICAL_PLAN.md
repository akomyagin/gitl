## 1. Вводные и ограничения

- Соло-разработчик, реализация с AI-ассистентом; приоритет — обучение Go
  (нагружать язык и stdlib, а не SDK).
- Целевые инфраструктурные расходы ≈ **$0/мес**; экономическая обоснованность
  необязательна.
- Требуется установленный системный `git` в `PATH` (для `os/exec`) — приемлемо,
  т.к. аудитория — разработчики.
- LLM обрабатывает **текст диффа** → жёсткий потолок по токенам → нужна стратегия
  усечения/чанкинга (см. §6, «Лимит токенов»).
- **Go 1.22+** (нужны `slices`, `maps`, `log/slog`, современный `errors.Join`).
- Кросс-платформенность: пути и конфиг через `os.UserConfigDir()`, а не хардкод.
- Без телеметрии и «звонков домой» — принципиально (доверие + приватность диффов).

## 2. Технический стек (§4.1 первоисточника)

| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | Go 1.22+ | Приоритет обучения; stdlib закрывает ~90% нужд |
| CLI | `spf13/cobra` | Стандарт индустрии (kubectl, gh, hugo), чистое дерево команд/флагов |
| Конфиг | `spf13/viper` | YAML + env + флаги в единой иерархии, интеграция с cobra |
| Git | системный `git` через `os/exec` | см. §4 |
| HTTP к LLM | `net/http` (stdlib) | Осознанно **без** SDK — тренировка ручного клиента, стриминга, retry |
| JSON | `encoding/json` (stdlib) | Парсинг ответов LLM и `--format=json` |
| TUI (opt, post-MVP) | `charmbracelet/bubbletea` + `lipgloss` | Только для `digest --tui`, поверх готовых артефактов |
| Логи | `log/slog` (stdlib) | Структурные логи, `--verbose` → debug |
| Тесты | stdlib `testing` + table-driven | Идиоматика Go; golden-файлы для рендеров |
| Сборка/релиз | `goreleaser` | Кросс-компиляция + GitHub Releases + Homebrew одним конфигом |

Сознательное ограничение зависимостей: git — stdlib `os/exec`, HTTP — stdlib
`net/http`. Внешние зависимости только там, где они дают обучающий каркас
(cobra/viper/bubbletea) или несоразмерны для ручной реализации (goreleaser).

## 3. Архитектура и структура проекта (§4.2)

```
gitl/

│   └── gitl/
│       └── main.go              # тонкий main: собирает root cmd, вызывает Execute()
├── internal/
│   ├── cli/                     # cobra-команды (по файлу на команду)
│   │   ├── root.go              # rootCmd, глобальные флаги, инициализация viper
│   │   ├── review.go
│   │   ├── changelog.go
│   │   ├── digest.go
│   │   └── version.go
│   ├── config/                  # загрузка/валидация конфига (viper → struct Config)
│   │   └── config.go
│   ├── gitlog/                  # чтение и парсинг git-истории
│   │   ├── runner.go            # обёртка над exec.Command("git", ...)
│   │   ├── parser.go            # парсинг --pretty/--name-status в структуры
│   │   └── types.go             # Commit, FileChange, Range, DiffStat
│   ├── llm/                     # HTTP-клиент к OpenAI-совместимому API
│   │   ├── client.go            # net/http, таймауты, retry+backoff
│   │   ├── stream.go            # разбор SSE-стрима (data: ... [DONE]) — post-MVP
│   │   └── offline.go           # эвристический offline-провайдер (без сети)
│   ├── prompt/                  # сборка промптов из шаблонов
│   │   └── templates.go         # embed через //go:embed
│   ├── render/                  # артефакт → md/text/json
│   │   └── render.go
│   └── tui/                     # bubbletea-модель для digest --tui (post-MVP)
│       └── digest_model.go
├── testdata/                    # golden-файлы, фикстуры git-вывода
├── .goreleaser.yaml
├── go.mod
└── README.md
```

**Почему `internal/`:** всё ядро в `internal/`, чтобы никто не импортировал
незрелый API как библиотеку — идиоматичный способ пометить «это приложение,
не библиотека». **`pkg/` намеренно не заводим на старте**: публичный API появится,
только если будет спрос («не строй библиотеку, пока её не просят»).

### 3.1 Поток данных (одна команда, §4.5)

```
cobra cmd → config.Load() → gitlog.Runner.Log/Diff(ctx, range)
   → prompt.Build(commits, diff, template)
   → llm.Client.Complete(ctx, prompt)  [retry/backoff, offline-fallback]
   → render.Render(artifact, format)   → stdout/file/TUI
```

Отмена по Ctrl-C — единый `context.Context` из `signal.NotifyContext`,
протянутый через exec и HTTP.

## 4. `os/exec` vs `go-git` — решение (§4.3)

**Выбор: системный `git` через `os/exec`.** Причины, релевантные цели
«написание + обучение Go»:

1. **Тренирует ядровой Go-навык** — работу с внешними процессами:
   `exec.CommandContext`, чтение `stdout`/`stderr` через `bytes.Buffer`/
   `bufio.Scanner`, коды возврата, контекст-отмена.
2. **Заставляет писать парсер** — самая обучающая часть: спроектировать
   `--pretty=format:` с разделителями, распарсить `--name-status`, покрыть
   table-driven-тестами на golden-выводе. С `go-git` данные приходят структурами —
   упражнение исчезает.
3. **Простая ментальная модель** — поведение 1:1 с терминалом; отлаживается
   тривиально (скопировал команду, запустил вручную).
4. **`go-git` тяжелее и с нюансами** — не 100% паритет с CLI по краевым случаям,
   больше зависимость.

Компромисс инкапсулируется: весь `os/exec` спрятан за интерфейсом в
`internal/gitlog` — если позже захочется `go-git`, меняется одна реализация,
команды не трогаются.

**Парсинг git log с control-разделителями (критичная деталь):**

```
git log --pretty=format:%H%x1f%an%x1f%aI%x1f%s%x1f%b%x1e --name-status <range>
```

- `%x1f` (ASCII **unit separator**, 0x1F) разделяет **поля** внутри коммита.
- `%x1e` (ASCII **record separator**, 0x1E) разделяет **коммиты**.

Control-символы из ASCII-зоны 0x1F/0x1E не встречаются в сообщениях коммитов,
поэтому надёжны как разделители. **Частая ошибка новичка — split по `\n`**: тело
коммита содержит и переводы строк, и пробелы, поэтому наивный split ломается.

Целевые типы (`internal/gitlog/types.go`):

```go
type Commit struct {
    Hash    string
    Author  string
    Date    time.Time
    Subject string
    Body    string
    Files   []FileChange
}
type FileChange struct {
    Status string // A, M, D, R...
    Path   string
    Old    string // для переименований (R)
}
```

## 5. Схема конфигурации (§4.4)

Два уровня, сливаются по приоритету
**флаг > env > `.gitl.yaml` (репозиторий) > `~/.config/gitl/config.yaml` (личный)**.
Личный файл — путь через `os.UserConfigDir()` (на Windows → `%AppData%\gitl`);
repo-level `.gitl.yaml` в корне репозитория коммитится в git и является общей
политикой команды.

```yaml
# ~/.config/gitl/config.yaml — личные настройки (секреты, провайдер по умолчанию)
llm:
  provider: "openai"          # "openai" | "ollama" | "azure_openai" — явный контракт
  api_key: ""                 # или env GITL_API_KEY (рекомендовано, особенно в CI)
  base_url: "https://api.openai.com/v1"
  model: "gpt-4o-mini"
  max_tokens: 1500
  temperature: 0.2
  timeout_seconds: 60
  max_retries: 3
  # azure_openai: endpoint, deployment, api_version — под-блок при provider=azure_openai
  # ollama: host (по умолчанию http://localhost:11434) — без api_key

cost:
  max_cost_usd: 0.50          # cost-guard: отказ выполнять запрос дороже порога
  warn_at_usd: 0.10           # предупреждение, но не блокировка

output:
  format: "md"                # md | text | json
  color: true

diff:
  max_diff_bytes: 120000
  exclude_globs:
    - "*.lock"
    - "*.min.js"
    - "vendor/**"
    - "*.svg"
```

```yaml
# .gitl.yaml — repo-level политика (коммитится в репозиторий, общая для команды)
policy:
  fail_on: "high"             # low | medium | high | never — порог гейтинга CI
  required_changelog_categories: ["Added", "Fixed", "Security"]
  exclude_globs: ["migrations/**"]   # ДОБАВЛЯЕТСЯ к личным exclude_globs, не заменяет

digest:
  repos:                      # мульти-репо digest
    - path: "../service-a"
    - path: "../service-b"
```

**Валидация при старте:** если `api_key` пуст/недоступен → offline-режим
с предупреждением в stderr (не падаем). Провайдер выбирается явным полем
`provider`, а не угадыванием по `base_url` — это и есть контракт
мультипровайдерности.

## 6. Разбивка по Этапам

> Каждый этап: **Фокус → Действия → Deliverable → Критерий успеха.**

### Этап 0 — Bootstrap *(этот этап)*

- **Фокус:** заложить репозиторий, окружение и документацию так, чтобы Этап 1
  начинался с готового каркаса, а не с пустого места.
- **Действия:**
  - структура репозитория, `.gitignore`, `LICENSE` (MIT) — уже есть;
  - документация в `docs/`: `PLAN.md`, `TECHNICAL_PLAN.md`, `POST_MVP_PLAN.md`
    (переработка `PLAN_SOURCE.md`);
  - `README.md` (краткий, человеко-ориентированный) в корне;
  - `CLAUDE.md` (модель-политика + git/dev-workflow) в корне;
  - `.claude/skills/go-cli-dev/SKILL.md` — конвенции проекта;
  - `docker-compose.yml` с одним сервисом `ollama` — dev-зависимость для теста
    мультипровайдерности (не часть продукта);
  - `go mod init github.com/akomyagin/gitl`;
  - минимальный каркас кода: `cmd/gitl/main.go` (тонкий main, печатает
    usage/version) + заглушки пакетов в `internal/*` с комментарием
    «реализация — Этап 1+»;
  - CI-скелет (GitHub Actions: `go build` + `go test`) — базовый workflow.
- **Deliverable:** репозиторий с полной документацией, каркасом кода
  и `go build ./...`, завершающимся успешно.
- **Критерий успеха:** `go build ./...` зелёный; структура каталогов
  соответствует §3; документация разбита на пронумерованные этапы.

### Этап 1 — Validation spike + каркас CLI *(недели 1–2)*

- **Фокус:** снять технический риск №1 (годится ли AI-ревью диффа в принципе)
  на одной захардкоженной команде **до** постройки CI/governance-контура,
  затем превратить прототип в настоящее CLI с governance-конфигом с первого дня.
- **Действия:**
  - Validation-first спайк: `gitl review HEAD~5..HEAD` — берёт диапазон, гонит
    `git log` + `git diff`, шлёт в LLM (или в offline-заглушку), печатает Markdown
    в stdout; захардкоженный промпт; примитивный `net/http`-вызов без retry.
  - `internal/gitlog` с `Log`+`Diff` через `os/exec`; парсер с control-разделителями
    (`%x1f`/`%x1e`); первые table-driven тесты парсера.
  - cobra (root + подкоманды) и viper; `internal/config` со struct + валидацией +
    слиянием `.gitl.yaml` → `~/.config/gitl/config.yaml` → env → флаги;
    глобальные флаги; `log/slog`.
  - детерминированный offline-провайдер.
- **Deliverable:** `gitl --help`, `gitl review <range>` с флагами и двухуровневым
  конфигом; осмысленное Markdown-ревью на реальном репозитории.
- **Критерий успеха:** вывод содержательный; парсер корректно ловит
  переименования/удаления; `go test ./...` проходит без ключа/сети; repo-level
  `.gitl.yaml` реально переопределяет личный конфиг с верным приоритетом.

### Этап 2 — LLM-клиент + мультипровайдерность + риск-скоринг + cost-guard *(недели 3–4)*

- **Фокус:** надёжный HTTP-клиент + явная поддержка трёх провайдеров; то, что
  превращает «текстовое ревью» в CI-гейтящий инструмент.
- **Действия:**
  - `internal/llm`: таймауты, `context`-отмена, retry с backoff+jitter,
    различение ретраебельных (429/5xx) и фатальных (401/400) ошибок через
    типизированные ошибки и `errors.Is`;
  - интеграционные тесты на retry через `httptest.Server`;
  - референс-конфиги для OpenAI-совместимого API, **Ollama** и **Azure OpenAI**
    (разные форматы auth/endpoint — вынести в `ProviderConfig`);
  - стратегия усечения диффа (`max_diff_bytes`, `exclude_globs`, явный маркер
    `[... diff truncated ...]`);
  - структурированный риск-скоринг (`low|medium|high`) в ответе `review`
    (промпт + парсинг + fallback-эвристика для offline);
  - `internal/render` с `schema_version` в JSON, задокументированная схема;
  - `--dry-run` (оценка стоимости по токенам до вызова) и `--max-cost-usd` guard.
- **Deliverable:** `internal/llm` с тестами на retry и тремя рабочими провайдер-
  конфигурациями; `gitl review --format=json` с валидным версионированным JSON
  и полем риска; `--dry-run` печатает оценку стоимости без реального вызова.
- **Критерий успеха:** мок-сервер 429→200 — клиент ретраит; прогон против Ollama
  локально и Azure OpenAI работают без правок кода, только конфигом;
  `--fail-on=high` даёт ненулевой exit code на рискованном синтетическом диффе;
  `--max-cost-usd=0.01` блокирует крупный дифф с понятным сообщением.

### Этап 3 — Полный набор команд + мульти-репо digest *(неделя 5)*

- **Фокус:** полный набор команд.
- **Действия:**
  - `changelog --since/--from/--to` (Keep a Changelog + conventional-commits);
  - `digest --days=N` с агрегацией по авторам/файлам/темам;
  - мульти-репо режим `digest --repos=a,b,c` — параллельный сбор через worker-pool
    (естественное обучение горутинам/каналам);
  - golden-тесты рендера.
- **Deliverable:** три команды, мульти-репо digest, три формата вывода
  (`md|text|json`).
- **Критерий успеха:** golden-тесты зелёные; digest по 3 тестовым репозиториям
  агрегируется корректно и не падает, если один из репо недоступен.

### Этап 4 — GitHub Action *(неделя 6)*

- **Фокус:** CI-интеграция как часть v1, не «когда-нибудь».
- **Действия:**
  - `action.yml` (composite или Docker action);
  - логика постинга `review` комментарием к PR через GitHub API;
  - выставление check status / exit code по риск-скорингу;
  - secrets-based передача ключа (`secrets.GITL_API_KEY`); документация
    безопасного использования в CI; тест на отсутствие ключа в stdout/stderr.
- **Deliverable:** Action, прогоняемый на собственном тестовом репозитории через
  реальный PR.
- **Критерий успеха:** PR получает содержательный комментарий-ревью; check status
  отражает риск-скоринг; ключ нигде не логируется.

### Этап 5 — Релизный инжиниринг: подпись, кросс-компиляция *(неделя 7)*

- **Фокус:** supply-chain-доверие — то, без чего команды не пускают сторонний
  бинарь в CI.
- **Действия:**
  - `.goreleaser.yaml` (linux/darwin/windows × amd64/arm64);
  - `cosign`/Sigstore keyless-подписание артефактов; `SHA256SUMS`;
  - `version` с build-provenance (commit hash + дата сборки через `-ldflags`);
  - semver-тег `v0.1.0`.
- **Deliverable:** подписанный GitHub Release с бинарями под 3 ОС.
- **Критерий успеха:** `cosign verify` на скачанном артефакте проходит; чек-суммы
  совпадают; свежескачанный бинарь запускается на чистой системе.

### Этап 6 — Marketplace + хардненинг + документация *(недели 8–10) — КОНЕЦ MVP*

- **Фокус:** товарный вид всего контура + надёжность на реальных диффах.
- **Действия:**
  - публикация Action в GitHub Marketplace (иконка, описание, версионирование
    через теги);
  - README/документация (usage, BYOK/мультипровайдер-настройка, `.gitl.yaml`-политика,
    offline-режим, cost-guard, Action-специфичная настройка: secrets, permissions);
  - расширенный набор фикстур/edge-cases (бинарные файлы, огромные моно-диффы,
    ребейзы, переименования); дополнительные unit/integration тесты;
  - dogfooding на нескольких реальных репозиториях автора, включая Action на PR.
- **Deliverable:** Action виден и устанавливаем из Marketplace; документация
  покрывает и CLI, и Action-сценарий; инструмент прогнан на реальных репо.
- **Критерий успеха:** сторонний человек подключает Action по README без вопросов;
  нет неожиданных падений на «грязных» реальных диффах; `-race` чист там, где есть
  конкурентность. **MVP считается завершённым.**

## 7. Риски и как их обойти (§6)

| Риск | Проявление | Обход |
|---|---|---|
| **Лимит токенов LLM на объёмном диффе** | Большой релиз не влезает в контекст → обрыв/ошибка/мусор | Порог `max_diff_bytes`; `exclude_globs`; стратегия «сначала `--stat` + сообщения коммитов, полный дифф — только для топ-N крупных файлов»; при переполнении — чанкинг по файлам с map-reduce агрегацией; явный маркер `[... diff truncated ...]`. |
| **Качество AI-сводки на шумных диффах** | Автоформатер/локи забивают ревью | `exclude_globs` по умолчанию (lock/min.js/vendor/svg); низкая `temperature` (0.2); промпт с инструкцией игнорировать форматирование; validation-first ловит это на Этапе 1. |
| **Недетерминизм → хрупкие тесты** | LLM-вывод нельзя проверить golden-файлом | Тестировать **всё, кроме** самого вызова LLM: парсер, рендер, retry (через `httptest`), сборку промпта. LLM-слой — за интерфейсом, подменяется offline/мок-провайдером. |
| **Кросс-платформенные грабли** | Пути/конфиг ломаются на Windows | `os.UserConfigDir()`/`filepath.Join` вместо хардкода; goreleaser собирает под все ОС; ранний smoke-тест бинаря под Windows (`--help`). |
| **Отсутствие `git` в PATH** | Паника/непонятная ошибка | Проверка `exec.LookPath("git")` при старте, дружелюбное сообщение с инструкцией. |
| **Утечка приватного диффа в LLM** | Пользователь шлёт закрытый код в чужой API | Документация в README; локальные/enterprise провайдеры (Ollama/Azure OpenAI) для резидентности; никакой телеметрии. |
| **Ложное доверие к AI-ревью при жёстком гейтинге** | `--fail-on=high` даёт false positive → блокирует мерж → инструмент выключают | Дефолт — **WARN, не BLOCK** (жёсткий gate — явный opt-in); консервативная калибровка риск-скоринга (Этап 2); ревью всегда показывает reasoning, а не только вердикт. |
| **Секреты в CI** | API-ключ утекает в логи/артефакты Action | Документация `secrets.*`-паттерна; ключ никогда не логируется даже в verbose; тест на отсутствие ключа в stdout/stderr при ошибках. |
| **Cost-guard не срабатывает** | Баг в оценке стоимости → незаметный крупный счёт | `--dry-run`/`--max-cost-usd` — отдельно тестируемая логика (unit-тесты на границы); консервативный дефолт; явное предупреждение при отключении guard. |
| **Неподписанный артефакт не пускают в CI** | Строгие CI-политики отклоняют неподписанные бинари | Cosign-подпись + `SHA256SUMS` — в MVP (Этап 5); задокументированная процедура верификации. |
| **Выгорание интереса** | Проект брошен на середине | Не решается урезанием скоупа — решается validation-first (Этап 1 даёт рабочее ядро рано) и чёткими per-этап deliverable с критерием «зелёно/не зелёно»; личная польза (dogfooding) — встроенная мотивация. |
| **Over-engineering ради «красоты»** | Абстракции, которых никто не просил | Правило «интерфейс появляется на второй реализации, не на первой»; `pkg/` не заводить, пока не попросят внешние потребители. |
