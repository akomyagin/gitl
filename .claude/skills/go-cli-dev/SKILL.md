# SKILL: go-cli-dev — конвенции проекта `gitl`

Конкретные конвенции **именно этого проекта** для написания Go-кода. Это не общий
гайд «как писать Go», а специфика `gitl`. Применяй при реализации любого этапа.
Опорные документы: [`../../../docs/TECHNICAL_PLAN.md`](../../../docs/TECHNICAL_PLAN.md),
[`../../../docs/PLAN.md`](../../../docs/PLAN.md).

---

## 1. Структура: `cmd/` + `internal/`, почему нет `pkg/`

- `cmd/gitl/main.go` — **тонкий**: только собирает root-команду и вызывает
  `Execute()`. Никакой бизнес-логики в `main`.
- Всё ядро — в `internal/`, чтобы Go-компилятор **запрещал** внешний импорт: это
  приложение, а не библиотека. Пакеты: `cli`, `config`, `gitlog`, `llm`, `prompt`,
  `render` (и `tui` в post-MVP).
- **`pkg/` не заводить.** Публичный API появится только при реальном спросе на
  использование как библиотеки. Правило: «не строй библиотеку, пока её не просят».
- По файлу на cobra-команду в `internal/cli/` (`review.go`, `changelog.go`,
  `digest.go`, `version.go`), общий каркас — в `root.go`.

## 2. Git-доступ: `os/exec`, не `go-git`

Решение зафиксировано (`TECHNICAL_PLAN.md` §4). Причина — обучающая: работа
с внешними процессами и **свой парсер**. Весь `os/exec` спрятан за интерфейсом
в `internal/gitlog` — если позже понадобится `go-git`, меняется одна реализация,
команды не трогаются.

- Вызовы — только `exec.CommandContext` (не `exec.Command`): контекст протягивается
  от `signal.NotifyContext` для отмены по Ctrl-C.
- Читать stdout/stderr раздельно (`bytes.Buffer`), проверять код возврата,
  оборачивать ошибку stderr'ом git.
- При старте проверять `exec.LookPath("git")` и давать дружелюбное сообщение,
  а не паниковать.

### Парсинг `git log` с control-разделителями

**Критично.** Формат:

```
git log --pretty=format:%H%x1f%an%x1f%aI%x1f%s%x1f%b%x1e --name-status <range>
```

- `%x1f` (ASCII **unit separator**, 0x1F) — разделитель **полей** внутри коммита.
- `%x1e` (ASCII **record separator**, 0x1E) — разделитель **коммитов**.

Почему так, а не пробелы/переводы строк: тело коммита (`%b`) содержит и `\n`,
и пробелы. Control-символы 0x1F/0x1E в сообщениях не встречаются → надёжны.

**Антипаттерн — split по `\n`.** Наивный `strings.Split(out, "\n")` ломается
на многострочных телах. Правильно: сначала split по `%x1e` (записи), потом
внутри записи split по `%x1f` (поля); блок `--name-status` парсить отдельно
(строки вида `M\tpath`, `A\tpath`, `R100\told\tnew`).

Целевые типы — `internal/gitlog/types.go`: `Commit{Hash,Author,Date,Subject,Body,Files}`,
`FileChange{Status,Path,Old}` (`Old` для переименований `R`).

## 3. LLM-клиент: retry + backoff (ручной `net/http`, без SDK)

`internal/llm/client.go` — вручную на `net/http`, осознанно без SDK (тренировка).

- Таймаут на запрос + общий `context`; отмена протянута насквозь.
- **Различать ошибки типизированно**: ретраебельные (429, 5xx, сетевые) vs
  фатальные (400, 401). Использовать `errors.Is`/типизированные ошибки, не строковое
  сравнение.
- **Retry с экспоненциальным backoff + jitter** (не фиксированная пауза — иначе
  thundering herd). Ограничение по `max_retries` из конфига.
- Секреты (`api_key`) **никогда** не логировать, даже в `--verbose`. Тест должен
  проверять отсутствие ключа в stdout/stderr при ошибках.
- Мультипровайдерность — через `ProviderConfig` (разные auth/endpoint для
  openai / ollama / azure_openai), выбор по явному полю `provider`.

### Offline-provider паттерн

`internal/llm/offline.go` — детерминированная эвристика **без сети**.

- Реализует тот же интерфейс, что и сетевой клиент → подставляется прозрачно.
- Активируется, когда `api_key` пуст/недоступен (с предупреждением в stderr,
  **не** падаем), а также принудительно для dev/CI.
- Детерминизм обязателен: один и тот же вход → один и тот же выход (иначе тесты
  становятся хрупкими). Для риск-скоринга — эвристика (например, по размеру диффа /
  ключевым паттернам), дающая стабильный `low|medium|high`.

## 4. Тесты: table-driven + golden

- **Всё, кроме самого вызова LLM, тестируется.** LLM-слой — за интерфейсом,
  в тестах подменяется offline/мок-провайдером.
- **Table-driven** для парсера (`gitlog`), сборки промптов, границ cost-guard.
- **Golden-файлы** в `testdata/` для рендеров (`render` → md/text/json). Обновление
  golden — через флаг (`-update`), сравнение — байт-в-байт.
- **`httptest.Server`** для retry-логики LLM-клиента (например, сценарий 429→200).
- Где есть конкурентность (worker-pool digest) — прогонять с `-race`.

## 5. `goreleaser` конвенции (Этап 5)

- `.goreleaser.yaml`: матрица linux/darwin/windows × amd64/arm64.
- Версия и build-provenance — через `-ldflags` (`main.version`, `main.commit`,
  `main.date`), команда `version` их печатает.
- Артефакты подписываются **cosign keyless** (Sigstore), прикладывается `SHA256SUMS`.
- SemVer-теги (`v0.1.0`); процедура `cosign verify` документируется в README.

## 6. Общие правила

- Go **1.22+**: можно `slices`, `maps`, `log/slog`, `errors.Join`.
- Кросс-платформенность: `os.UserConfigDir()` / `os.UserCacheDir()` / `filepath.Join`,
  **никогда** не хардкодить `~/.config`.
- Логи — `log/slog`; `--verbose` поднимает уровень до debug.
- Интерфейс вводится **на второй реализации**, не на первой (анти-over-engineering).
- Каждая заглушка `internal/*` в Этапе 0 помечена комментарием «реализация — Этап N+»;
  при реализации заменять содержимое, а не плодить параллельные файлы.
