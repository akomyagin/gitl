# CLAUDE.md

Инструкции для AI-сессий (Claude Code) в репозитории `gitl`. Эти инструкции
имеют приоритет над поведением по умолчанию — следуй им точно.

## Что это за репозиторий

`gitl` (git-log-lens) — CLI/CI-инструмент на **Go**: AI-ревью git-истории
с машиночитаемым риск-скорингом для гейтинга в CI, changelog, мульти-репо digest,
GitHub Action-обёртка. Соло pet-проект с приоритетом **обучения Go** и сознательно
поднятой продуктовой планкой (CI-ready, не «личный скрипт»). Расходы ≈ $0/мес,
BYOK + мультипровайдерность, без телеметрии.

Полное видение — [`docs/PLAN.md`](docs/PLAN.md); технический план и разбивка по
этапам — [`docs/TECHNICAL_PLAN.md`](docs/TECHNICAL_PLAN.md); за пределами MVP —
[`docs/POST_MVP_PLAN.md`](docs/POST_MVP_PLAN.md); первоисточник — `docs/PLAN_SOURCE.md`.

## Структура репозитория

| Путь | Содержимое |
|---|---|
| `docs/` | `PLAN.md`, `TECHNICAL_PLAN.md`, `POST_MVP_PLAN.md`, `PLAN_SOURCE.md` (первоисточник) |
| `.claude/skills/go-cli-dev/SKILL.md` | Конвенции именно этого проекта для написания кода |
| `cmd/gitl/` | Тонкий `main.go` — собирает root cmd, вызывает `Execute()` |
| `internal/` | Всё ядро: `cli/`, `config/`, `gitlog/`, `llm/`, `prompt/`, `render/` (и `tui/` в post-MVP) |
| `docker-compose.yml` | **Только dev-зависимость** — сервис `ollama` для локального теста мультипровайдерности; сам `gitl` в контейнер не оборачивается |

`pkg/` намеренно **нет** — всё ядро в `internal/`, чтобы никто не импортировал
незрелый API как библиотеку. Публичный API появится только при спросе.

## Модель-политика (какую модель на что вызывать)

Зафиксировано пользователем. Придерживайся строго:

- **Sonnet 5** — анализ документации; основной флоу чата (диалог с пользователем
  по умолчанию). Это модель, ведущая сессию.
- **Opus 4.8** — написание и планирование, глубокие рассуждения, ревью.
  Вызывается через **Agent-тул с `model: "opus"`** для планирования нового этапа
  и для независимого ревью написанного кода.
- **Fable 5** — написание кода. Вызывается через **Agent-тул с `model: "fable"`**
  для реализации конкретной задачи/этапа.

## Git / dev-workflow (обязательный процесс на каждый этап/задачу)

Главная ветка — **`master`** (не `main`).

1. **Новая ветка** от `master` на этап/задачу.
2. **Анализ и подготовка** — основной чат (**Sonnet 5**).
3. **Написание кода** — Agent с **`model: "fable"`**.
4. **Просмотр результата** — основной чат (**Sonnet 5**).
5. **Независимое ревью** — Agent с **`model: "opus"`**.
6. **Тестирование** — `go test ./...`.
7. **Финальный просмотр + актуализация документации** в `docs/` — **Sonnet 5**.
8. **По завершении этапа:** commit, push ветки, создание **PR в `master`**.

Коммить/пуш только когда этап завершён (или пользователь явно попросил).

## Команды разработки

```bash
go build ./...                 # сборка (должна быть зелёной с Этапа 0)
go test ./...                  # тесты
go run ./cmd/gitl version      # запуск (пока version/usage; команды — с Этапа 1)
go run ./cmd/gitl --help

docker compose up ollama       # локальный Ollama для теста мультипровайдерности (dev-зависимость)
```

Перед коммитом кода: `go build ./...` и `go test ./...` должны проходить;
желателен `gofmt`/`go vet`.

## Ключевые архитектурные решения (не переоткрывай их без причины)

- Git-доступ — системный `git` через `os/exec` (не `go-git`): обучающий эффект +
  свой парсер. Парсинг `git log` — с control-разделителями `%x1f`/`%x1e`,
  **не** split по `\n`. Подробности — `docs/TECHNICAL_PLAN.md` §4 и SKILL.md.
- LLM-доступ — ручной `net/http`-клиент **без SDK** (тренировка retry/стриминга).
- Мультипровайдерность — явное поле `provider` (openai/ollama/azure_openai),
  не угадывание по `base_url`.
- Offline-режим — детерминированная эвристика без сети (dev/CI без ключа + fallback).
- Правило зависимостей: «интерфейс появляется на второй реализации, не на первой».
